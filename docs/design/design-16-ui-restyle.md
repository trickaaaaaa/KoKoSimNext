# 設計書16: UIリスタイル「テロップと電光掲示板」（書体3役＋甲子園ラインスコア）

作成: 2026-07-21。承認済みの方向性（B案「新聞とテロップ」＋甲子園球場の電光掲示板をシグネチャに採用）を実装計画に落としたもの。
**まだ実装していない。** 着手条件・順序は §6 を参照。

---

## 0. 背景（なぜやるか）

現行UIの「AI臭さ」を診断した結果、原因は以下の4点に集約された（2026-07-21、スクショ5画面＋tokens.uss＋フォントアセットの実地調査）:

1. **書体**: ヒラギノ丸ゴ Pro W4 の1書体・1ウェイトのみ、太字はすべて疑似ボールド（`-unity-font-style: bold`）。
   数字も本文と同じ丸ゴで、UI原則③「数値はコンデンス書体」が未実装。丸ゴ＝優しいアプリの声で野球の世界観と不一致
2. **カード文法**: 「暗い面＋単一アクセント＋同型角丸カードの均等グリッド」はAI生成UIの定番構造そのもの。
   特にホームは同じ形・同じタイトル位置のカードが等間隔に並び主役がいない
3. **コンセプトが言葉止まり**: 「スコアボード緑×アンバー」というテーマ名に対し、視覚要素としての
   スコアボードらしさ（ドット数字・イニング枠・ランプ）がどこにもない
4. **装飾語彙**: 英語eyebrow（「TOURNAMENT PREVIEW ――」）、ピル型チップ、丸端バー等の
   2023-24年Tailwindダッシュボード語彙

さらに実務上の決定打として、**ヒラギノはmacOS同梱ライセンスでありゲームへの埋め込み配布は不可**
（SDFアトラス化しても字形埋め込みに変わりなし）。フォント差し替えは出荷までにどのみち必須。

**守る資産（AI臭の原因ではない・変えない）**: 情報密度・行高・整列の規律、ランク色システム（S〜G色相ランプ＋文字併記）、
日本語コピーの具体性、試合2D盤面の視覚言語。触るのは声（書体）と手触り（面・形・語彙）であって骨格ではない。

## 1. コンセプト: 二つの声のルール

甲子園のTV中継の画作りを設計原理にする。**画面には「球場の中の情報（電光掲示板）」と
「放送側の情報（テロップ・新聞）」の2つの世界が同居し、それぞれ固有の声を持つ。**

| 世界 | 対象 | 声（書体・面） |
|---|---|---|
| 掲示板の中 | スコア・B/S/Oカウント・イニング・週カウントダウン等「球場が掲示する情報」 | ドット文字（DotGothic16）、黒に近い面、枠グリッド、ランプ |
| 掲示板の外 | 校名・大会名・見出し（中継テロップ／新聞の語彙） | **太明朝**。「校名は常に太明朝」が本リスタイルのシグネチャ規則 |
| データの地 | 表・本文・ラベル | 中立サンセリフ＋**コンデンス数字**（FM型密度の担い手） |

境界規則: 掲示板部品（ScoreboardStrip / LineScorePanel）の中はすべてドット文字。外に出たら明朝＋サンセリフ。
この境界があるため2つの語彙が同一画面に同居しても混ざらない。

**注意**: 掲示板の視覚言語を借りるのは「見た目」であって「情報」ではない。
試合スコアという情報が出るのは試合画面のみ（ホーム等のヘッダーに出すのは週・カウントダウン等の既存情報だけ）。

## 2. 書体計画（3役＋掲示板。すべてGoogle Fonts・OFL＝埋め込み配布可）

**2026-07-21 のF0ゲートで確定済み**（検証ボードの実物を見て決定）。以降この表を正とする。

| 役割 | 確定書体 | USSクラス | 用途とルール |
|---|---|---|---|
| ディスプレイ（明朝） | **Shippori Mincho B1 Bold＋ExtraBold**（併用） | `.f-display-lg`（Bold）／`.f-display`（ExtraBold） | 校名・大会名・ページ見出し。**16px以上・太ウェイト限定**（暗背景で細い横画が潰れるため本文使用禁止）。**32px以上の大見出しだけ Bold**、16〜24pxは横画が残る ExtraBold を使う（＝`-lg` の方がウェイトは軽い。名前はサイズ用途を表す） |
| 本文・UI | **IBM Plex Sans JP** Regular/Medium/SemiBold | `.f-body` / `.f-body-md` / `.f-body-bd` | ラベル・本文・ボタン。実ウェイト3段で**疑似ボールド全廃** |
| データ数字 | **Oswald** Medium/SemiBold | `.f-num` / `.f-num-bd` | 表・打率・金額・背番号。欧文専用＝和文グリフを持たないので、和文と混ぜる行は Label を分ける |
| 掲示板 | **DotGothic16** | `.f-dot` | 掲示板部品の中専用。**16の整数倍サイズ限定**（`--fs-dot-16/32/48`）。非整数倍でドット感が消えることはF0で実証済み |

不採用: Zen Old Mincho（小サイズで最も先に細る）／Murecho／Zen Kaku Gothic New（11pxで薄い）／Barlow Semi Condensed。
DSEG（7セグ）はGoogle Fontsに無く未取得。DotGothic16 は和文もドットで出せるため単独で足りると判断した。

### 技術方針

- TTFは `Assets/UI/Fonts/` に配置し、**OFLライセンスファイルを同梱**する
- ウェイトごとにSDFフォントアセットを作成。SDFAA構成・整数スケール・Retina等倍キャプチャの既知手順に従う
  （メモリ「UI文字鮮鋭化」「パネルスケールと滲み」参照）
- 和文フォールバック: 欧文数字フォント（Oswald等）→本文和文フォントのフォールバックを設定（豆腐対策）。
  ただし混植の基本は既存慣行どおりLabel分割（`sb2-cell__k` / `__v` 方式）で対処
- フォント指定は `var()` ではなく**ユーティリティクラス**で行う（`.ui-root` と同じパターン。
  USS変数にアセット参照を入れる方式は採らない＝既知のvar系の罠を回避）

## 3. トークン／クラス設計（tokens.uss 追加分）

```css
/* 書体3役＋掲示板（ユーティリティクラス。-unity-font-definition を直接指定） */
.f-display { ... }   /* 太明朝: 校名・大会名・ページ見出し */
.f-body    { ... }   /* サンセリフ Regular（.ui-root の既定を置換） */
.f-body-md { ... }   /* サンセリフ Medium */
.f-body-bd { ... }   /* サンセリフ SemiBold（旧 -unity-font-style:bold の代替） */
.f-num     { ... }   /* コンデンス数字 */
.f-dot     { ... }   /* ドット（掲示板部品内専用） */

/* 掲示板専用サイズ（16の整数倍のみ） */
--fs-dot-16: 16px;  --fs-dot-32: 32px;  --fs-dot-48: 48px;
/* 掲示板面（--sf-base より暗い黒緑。値はF1で確定） */
--sb-face: #0A0F0C;
```

- 移行完了後、`-unity-font-style: bold` は原則禁止（USS内grepで検出できる状態にする）
- 角丸は縮小方向で再検討（掲示板・新聞の語彙はどちらも直線的）。最終値はF1で確定

## 4. 部品辞書への追加（UI-BUILD-METHOD Step 2 準拠・画面より先に部品を作る）

1. **ScoreboardStrip** — `Components/TopBar.uxml` 中央セル（現在／夏予選まで）の電光掲示板化。
   黒面＋ドット数字。アンバーは点灯箇所（カウントダウン等の最重要値）のみ、基本は白ドット。
   **表示情報は現状と同一**（週・カウントダウン。スコアは出さない）。TopBarは全画面単一ソースなので1改修で全画面波及
2. **LineScorePanel** — 試合画面専用の甲子園式ラインスコア。校名・イニング得点1〜9（＋延長）・R H E・
   B/S/Oランプ。ランプ色は現行ルール準拠（B=緑/S=黄/O=赤）＋文字併記で色覚対応。
   MatchLive上部の現行スコアバーを置き換える。本リスタイルの本丸
3. **SchoolName** — 太明朝校名＋都道府県/シードの小書き。展望・ブラケット・対戦カード・見出しで統一使用
4. **NumCell** — コンデンス数字の右揃えセル。全テーブルの数値列がこれを使う（列幅トークンは既存を踏襲）

## 8. リスクと対策

| リスク | 対策 |
|---|---|
| 明朝の小サイズ潰れ（暗背景×細横画） | 16px以上・太ウェイト限定をクラス設計で強制。本文は絶対にサンセリフ |
| DotGothic16の非整数倍濁り | サイズトークンを16/32/48に固定 |
| 和文動的アトラスの初回生成ヒッチ | F0で解消済み。Static＋`data/`走査の事前ベイクにした（1615字・未収録0） |
| **静的アトラスの焼き漏れ＝豆腐** | 2026-07-21 に実発生。旧 KokoSimFont は Static 化時点で動的生成済みの667字だけが固定され、`data/` が要求する1206字のうち**758字が欠落**（基本ひらがなの「ぎ・ご・づ・ど・ぬ・ふ・ほ・ぼ・む・ゃ・ゅ・よ・ろ」を含む）。対策＝文字セットを手書きせず `data/` から自動収集し、語彙追加時は `FontAssetBuilder` を回す |
| アンバー希少性ルール（UI原則②）との衝突 | 掲示板内は白ドット基本、アンバーは点灯ランプと最重要数値のみ |
| 欧文数字フォントの和文混入で豆腐 | フォールバック設定＋Label分割の既存慣行 |
