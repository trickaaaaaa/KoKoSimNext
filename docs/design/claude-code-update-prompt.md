設計を大きく更新しました。前回実装した内容に対して、既存コードの修正を伴う仕様変更と、新規システムが多数あります。

まず `docs/design/CHANGELOG-2026-07-16.md` を読んでください。本日の差分が
🔴BREAKING（既存コード要修正）/ 🟢NEW（新規実装）/ 🟡BALANCE（後で調整）に分類されています。
最新の完全な設計は docs/design/design-01〜11 と data/pref-formats/ が正です。

進め方（この順で、段階的に。各ステップでビルド・テストを通してから次へ）:

1. **現状把握**: CHANGELOG の🔴BREAKING項目について、前回実装したコードの該当箇所を洗い出し、
   「どのファイル・クラス・メソッドが影響を受けるか」を先にリストアップして提示してください。
   まだコードは変更せず、影響範囲の見取り図を出すこと。

2. **🔴BREAKING を先に反映**（データ構造・生成・計算の作り直し）。特に順序に注意:
   - 選手データモデルの変更（走塁系パラメータ追加、肩→肩+送球精度、伸びしろ、守備位置適性、投手経歴フラグ）
   - 選手生成の作り直し（運動能力ベースライン2段階、投打の左右条件付き、球速二層分布、球種ランク=ストレート含む、新入生の総合力→凸凹配分）
   - 走塁/盗塁/バント/スクイズ解決式を新パラメータ参照へ。走塁系スキルの削除
   - 各ステップ後に既存のヘッドレステスト（設計書07のDoD）を回し、統計が破綻していないか確認

3. **🟢NEW を追加**（新規システム）。依存の少ない順で:
   - 調子システム / 試合限りの好不調 → 打席解決に補正フックを追加
   - 怪我システム / 選手成長・気づきイベント / 監督成長イベント
   - **試合エンジンの出力形式**: プレーを「タイムライン付きイベント列（座標＋時刻）」で出力する契約に
     （CHANGELOG 32-33）。既存エンジンが最終結果のみ返すなら、タイムライン出力へ拡張。
     守備連携は「守備陣形ルール表」で自動生成。UIは物理計算せず時刻tで再生するだけ
   - 設計書09（采配: サイン・伝令・DH・タイブレーク・主将）
   - 設計書10（スキル: data/skills.yaml 駆動、有無フラグ）
   - 設計書11（敵AI: 三層。プレイヤーと同じ采配システムを流用）
   - 大会まわり（設計書05）: pref-formats のYAMLローダ、県内地区、センバツ選考、現代ルールトグル
   - **UI画面は該当フェーズで**（下記UI方針参照）

3b. **UIの扱い（重要）**:
   - 設計書07のロードマップ通り、Phase5まで純C#エンジンをCLIで検証。**UIとエンジンを混ぜない**
   - UI技術は Unity UI Toolkit（UXML/USS）
   - 試合ビューは「エンジンが出力するタイムラインをUIが再生するだけ」構造（CHANGELOG 32-35）。
     参照モック mock-match-2d-view.html / mock-steal-2d-view.html のデータ構造・9人守備の描き分けに従う
   - 選手・成績・大会プレビュー等の画面は Claude Design モックが視覚的な正（CHANGELOG 36-38）。
     HTMLはそのまま使わず、レイアウト・配色・コンポーネントを UXML/USS に翻訳する参照仕様として扱う

4. **🟡BALANCE はプレースホルダ**で構わない。data/ のYAML係数として外出しし、
   ヘッドレス1万試合の検証（設計書07）で後から調整できる形にしておくこと。

制約・原則（設計を貫くもの。実装時も厳守）:
- 打席解決パイプライン（設計書01 §2）の構造は変えない。調子・怪我・スキル等は「補正係数」として差し込む
- パラメータ/精神力システムで表現できるものをスキルや別能力にしない（二重計上禁止）。
  特に「対左右」「チャンス/逆境/対ピンチ」を能力やスキルとして復活させないこと
- プレイヤーとAIは同じ采配システムを使う（AI専用の裏ロジックを作らない）
- data駆動を徹底（skills.yaml / pref-formats/*.yaml / バランス係数）。Claude CodeがYAML編集で調整できるように
- Unity非依存の純C#エンジンを維持（Phase5までCLI検証、設計書07のロードマップ準拠）
