# UI構築手法 — トークン＋部品辞書方式（画面非依存の正典）

このドキュメントは **UIをどう作るか** の方法論を定める。個別画面の仕様（何を表示するか）は
設計書06および各機能設計書に、**作り方**はここに集約する。以降のUI実装はすべてこの手順に従う。

> **判断基準（何を目指すか）は `CLAUDE.md`「UI原則」7箇条を参照。** このドキュメント（作り方＝手順）と
> UI原則（判断基準＝FM型の密度・強調の希少性・数字が主役…）は補完関係で、UI作業では常に両方に従う。

## 背景（なぜこの方式か）

初期に採った「Claude Design の HTMLモックを見た目の正として UXML/USS へ機械移植する」方式は失敗した。
理由は2つ:

1. **USSはCSSの完全互換ではない**（`gap` / CSS grid / 擬似要素 などが非対応）。機械移植すると
   欠落機能でレイアウトが崩れ、崩れを目視で追う無限ループに入る。
2. **ヘッダーとデータで幅をベタ書きすると横ずれが累積する**。片方をいじると崩れ、原因が追えない。

そこで方針を転換する:

> ** トークン（数値）と部品（UXML/USS）が唯一の「正」。HTMLモックは参照専用。**
> **以降のUIは、トークンの範囲内で部品を組み合わせるだけで作る。**

数値と部品が正なので、モックが多少ずれても部品が揃っていれば画面は破綻しない。

---

## 大原則（不変条件・違反しない）

1. **モックは参照専用**。HTML/CSS を UXML/USS へ機械移植しない。見た目の意図だけを読み取り、
   崩れない範囲を部品化する。
2. **辞書が唯一の正**。色・サイズ・余白・角丸をトークン外で直書きしない。新しい見た目が必要になったら
   **画面側で自作せず、部品辞書に追加してから使う**。
3. **全部Flexbox**。USS非対応記法（grid / 擬似要素 / `gap` / `align-items: baseline` 等）は使わない。
4. **エンジン純度の維持**。UIはエンジン出力（タイムライン／ViewModel）を表示するだけ。ロジックを持たない。
5. **人間の目視は最終段のみ**。部品修正→スクショ出力→自分で画像確認、を機械で回してから人に見せる。

---

## Step 1: デザイントークン（`Assets/UI/tokens.uss`）

USSカスタムプロパティ（`--var`）で、以下を**1ファイルに集約**する。既存の `KokoSimTheme.uss` の
色定義はここへ吸い出し、テーマ／各画面USSは `var(--...)` を参照するだけにする。

### 定義する軸

- **カラー**: スコアボード緑系（`--color-field` #14231B / `--color-panel` #1D3227 /
  `--color-panel-deep` #0E1A13 / `--color-line` #38543F）、チョーク白 `--color-chalk` #EFF4EA、
  副次 `--color-muted` #9DB3A0、アンバー `--color-lamp` #F5C64A、警告 `--color-warn` #E86A4A、
  ランクS〜Gの7色（`--rank-s` … `--rank-g`、色覚対応・文字併記前提）。
- **タイポグラフィ**: フォント指定（SDF丸ゴ `KokoSimFont`）と、見出し／本文／数値の**サイズ段階**
  （例 `--fs-page` 18 / `--fs-body` 16 / `--fs-label` 13 / `--fs-cap` 11 / `--fs-num-lg` 22）。
  フォントは基底クラス `.ui-root`（`tokens.uss`）に `-unity-font-definition` で1回だけ定義し、
  **各画面はルート要素に `class="ui-root"` を必ず付ける**（下記「文字がガビガビ」参照）。
- **スペーシング**: 余白の段階（`--sp-1` 4 / `--sp-2` 8 / `--sp-3` 12 / `--sp-4` 16）。モックから実測。
- **角丸・ボーダー**: `--radius-chip` 6 / `--radius-panel` 10、`--border-line` の色と太さ。
- **レイアウト寸法（重要）**: テーブル列幅など、**ヘッダーとデータが同じ変数を参照すべき値**を
  トークン化する（`--col-id` 220 / `--col-grade` 56 / `--col-preset` 120 …）。これが横ずれの構造的対策。

### 記法

```css
:root {
    --color-field: #14231B;
    --col-id: 220px;
}
.tp-th--id, .tcell--id { width: var(--col-id); }   /* ヘッダーとデータが同一変数を参照 */
```

> 注: USSの `var()` はUnity 6で使用可。ただしCSS変数の全機能（`calc`との併用等）は揃わない。
> 迷ったら「変数で数値を一元化する」目的だけに使い、凝った演算はしない。

## Step 2: 部品辞書（`Assets/UI/Components/<Name>.uxml` ＋ USSクラス）

**1部品＝1ファイル**。再利用される見た目の最小単位をUXMLテンプレート＋USSクラスとして実装する。
C#から差し込むデータは、UXMLの `name` 属性で明示的に受け渡す（Controllerが `root.Q("...")` で参照）。

### 種の部品セット（辞書は育てる。最初はこの範囲で足りる）

| 部品 | 役割 |
|---|---|
| `RankChip` | 能力/総合ランクバッジ（S〜Gの7色・文字併記・色覚対応） |
| `NumberBadge` | 背番号バッジ（丸型／エース・正捕手の慣例色） |
| `StatBar` | 横バー（使用率・疲労・バジェット等。色はトークン参照） |
| `SegmentedControl` | N段トグル（強度 軽/普/猛 など。選択段ハイライト） |
| `Chip` | 小ラベル（配分チップ・フィルタチップ・タグ） |
| `Switch` | オン/オフトグル（委任など） |
| `StepperRow` | アイコン＋名称＋(−値+)のステッパー行（メニュー配分など） |
| `Panel` | 見出し付きの区切り枠（画面の基本容器） |
| `Banner` | 強調告知バー（合宿など） |
| `Row` | 一覧の1行（複数部品を横に組む器） |

各部品は**サンプルデータ入りのプレビュー**（UI Builder のPreview、またはStep 3の単体スクショ）を持つ。

> 既存の `TrainingPlan.uss` に作ったインラインクラス（`.grade` `.trow` `.ubar` `.seg` `.mslot` `.switch`
> `.tp-camp` 等）は、事実上この辞書の初稿。作り直しでは**これらを名前付き部品として抽出・整理する**
> （ゼロからではなく抽出リファクタ）。

## Step 3: スクショ検証の自動化（`screenshots/` へPNG出力）

目視の撮り直しループを排除するため、**部品／画面を決定論的にPNG化するエディタスクリプト**を用意する。

### 方式（GameViewのキャプチャに依存しない）

UI Toolkit の UIDocument はGameViewの再描画に依存し、フォーカスや描画タイミングで空フレームになる
（実地で確認済み）。よって GameView をキャプチャせず、**パネルを RenderTexture に描いて `ReadPixels`** する。
これは headless / batchmode でも決定論的。

recipe（エディタ静的メソッド）:
1. 対象UXMLを `panelSettings` 付きの UIDocument に読み込む。
2. `panelSettings.targetTexture = rt`（任意解像度のRenderTexture）を割り当てる。
3. レイアウトを1パス進める（`panel` にアタッチ → 数フレーム `QueuePlayerLoopUpdate`）。
4. `RenderTexture.active = rt; tex.ReadPixels(...); File.WriteAllBytes(png)`。

起動例:
```
Unity -batchmode -projectPath . -executeMethod UIScreenshot.Capture -uxml PlayerRow -quit
```

### 運用ループ

> 部品を修正 → スクショ出力 → **自分で画像を確認** → 直す、を人手を介さず回す。
> 人間の目視確認は最終段（画面が組み上がってから）だけにする。

## Step 4: 画面の組み立て（1画面ずつ）

- 画面は**辞書の部品の組み合わせ＋トークンの範囲**だけで構成する。新しい見た目が要るなら、
  画面側で自作せず**先に辞書へ部品を追加**してから使う。
- 完成後、HTMLモックと並べたスクショ比較を出力し、差分（意図的な簡略化を含む）を箇条書きで報告する。

## 共通シェル：トップバー（全画面で同一・単一ソース）

**上部ヘッダーとナビタブは画面ごとに作り直さない。** 全画面が共通の
`Assets/UI/Components/TopBar.uxml`（スコアボード `sb2` ＋ ナビ `nav`）を**単一ソース**として読み込む。
タブを切り替えても上部とタブのデザインが変わってはいけない（＝ホーム画面の見た目に統一）。

- 各画面UXMLの冒頭で宣言し、ルート直下に1回だけ差し込む:
  ```xml
  <ui:Template name="TopBar" src="Components/TopBar.uxml" />
  ...
  <ui:Instance template="TopBar" />
  ```
- **アクティブタブの強調（`nav__item--on`）は `ScreenRouter` が付ける**。TopBar 側に直書きしない
  （直書きすると全画面で同じタブが光る）。ナビのクリック配線も `ScreenRouter.Show` が担う。
- スコアボードの動的値（週・チーム総合力ランク・部費・名声・信頼度）は**各画面のコントローラが
  `name` で埋める**（未実装値は TopBar.uxml の既定表示＝Phase 1 のプレースホルダで可）。
- 「今週を進める」ボタン（`name="advance"`）は各画面のコントローラが配線する
  （ホーム＝週送り実処理、練習＝プレビュー週送り 等）。

## モーダル／オーバーレイ（同一ドキュメント内・display開閉・要 ui-root）

**別UIDocument／別PanelSettings でモーダルを作らない。** `ScreenRouter` は複数 UIDocument が同一
PanelSettings に同居するとレンダリング競合でパネルごと消えるため、**SetActive で常に1ツリー**に保つ設計。
モーダルはその制約に従い、**対象画面のドキュメント内オーバーレイ VisualElement** として実装する。

- 部品辞書（`components.uss`）の `.modal-scrim`（全画面 `position:absolute` の暗幕）＋ `.modal-card`
  （中央カード）を使う。スクリムは画面ルート（`.tp-root` 等）の**末尾の子**に置き、`toast` と同じく
  `style.display`（None↔Flex）で開閉する。開くとき `element.BringToFront()` を呼ぶ。
- **スクリムのルート要素にも `class="ui-root"` を必ず付ける**（フォント継承。付け忘れると文字がガビガビ）。
- 中身は動的生成でよいが、`name` ホストは UXML に置く。開閉フラグは各画面コントローラが持ち、
  `Render()` で `display` と中身を一括更新する（トグル・±調整などの操作後も開いたまま最新化される）。
- 実例＝**練習設定の統合モーダル**（`TrainingPlan.uxml` の `tp-modal`）を1枚に統合:
  **基本は「自分で配分」（10分単位のメニュー行＋守備ポジション行を同じ mslot 粒度で縦並び）を主**に置き、
  **お任せプリセット（説明＋「伸びる能力」相対バー）は任意の選択肢として後段**に置く。右パネルに
  「この設定で伸びる能力」（プリセット選択・±調整で即時更新）＋残バジェット。**打撃〜投手の全能力を常に一画面に列挙**し、
  バーは「今週の成長量（進んだレベル進捗）」で効果のある能力だけ伸び、伸びない能力も淡色＋「—」で必ず出す（投手能力も野手能力も常に見える）。
  行はクリックでこのモーダルを開くだけにし、テーブルは「誰・総合・現在の設定＋主眼メニュー・複製」に絞って全幅表示する
  （使用率 300/300・％や見込み列は行に出さない＝すっきり。設定バーは行幅いっぱいに広げて余白を埋める）。
  並びは**学年＋固定ID順**を既定にしてレギュラー変動で入れ替わらないようにする（背番号＝強さ由来で不安定なため既定に使わない）。
  練習時間の刻みは engine 単一ソース `TrainingPresets.StepMinutes`（=10分）で、プリセット配分もこの倍数に量子化する。
  休養は明示メニューにせず「残り（未配分）」で表す。

## USS／UXMLの落とし穴（実地で判明・作り直しで踏まない）

- **UXMLコメントに `--`（二重ハイフン）を書かない**。XML仕様でコメント内に `--` は不可。書くと
  ファイル全体がパース失敗し、**VisualTreeAsset が空でインポートされ**（`CloneTree()` の childCount=0）、
  `<ui:Instance>` が何も描画しない。説明文に `nav__item--on` 等のクラス名を書くときは要注意
  （`nav__item` の修飾子、と言い換える）。コンソールに `An XML comment cannot contain '--'` が出る。
- **`<ui:Template src>` は他画面と同じ相対パス**でよい（`Components/TopBar.uxml`）。空表示の真因は
  ほぼ上記コメント破損か、参照先UXML自体の不正。まず `CloneTree().childCount` を実測して切り分ける。
- **`gap` 非対応** → `margin` で間隔を作る。
- **CSS grid 非対応** → `flex-wrap` で段組みを表現。
- **`align-items: baseline` 非対応** → `flex-end` 等で代用。
- **固定高セクションは `flex-shrink: 0`** を付ける。付けないと縦方向で潰れて `NaN` レイアウトになる。
- **ヘッダーとデータの幅は同一トークンを参照**（Step 1）。別々にベタ書きすると横ずれが累積する。
- **固定幅の列（flex子要素）には必ず `flex-shrink: 0`**。付けないと、同じ行にある `flex-grow` 列（例:
  配分列）の**中身が広い行だけ** flexbox が固定列から幅を奪い、`--col-*` を共有していても列がずれる。
  相方の `flex-grow` 列は `flex-basis: 0; min-width: 0` にし、中身で希望幅が膨らまないようにする
  （`flex-basis: auto` は中身の最大幅が基準になり、折り返し前提の多チップ行で行が肥大→他列を圧縮する）。
  これは Step 1 の「列幅トークン共有」を成立させるための必須条件（横ずれの真因はほぼこれ）。
- **既定 Label の margin/padding で見出しがずれる**。幅を持つ見出しは `<Label>` を使うと既定テーマの
  margin（約 2.4/3.6px）が列間に累積し、`margin:0; padding:0` の `<VisualElement>` データ列と食い違う。
  見出しセルに `margin: 0; padding: 0` を付けて列境界を一致させる。
- **ScrollView の縦スクロールバー分だけ、中の行がヘッダーより狭い**（既定テーマで 24px）。ヘッダーを
  ScrollView の外に置く表では、`flex-grow` 列にこの差が全部乗って以降の列がずれる。ヘッダー右端に
  スクロールバー同幅（`--scrollbar-w`）のガター要素を置き、`flex-grow` 列を行内容と同幅にして揃える。
- **絵文字グリフはSDFフォントに無い**ことが多い → アイコンは漢字1文字や記号（フォント収録字）を使う。
- **`translate` のパーセント併用**は動くが凝らない。位置合わせは基本Flexで。
- **文字がガビガビ（ジャギる）＝SDFフォント未適用**。PanelSettings 既定テーマの legacy ビットマップ
  フォント(NotInter)が使われている状態。`-unity-font-definition` は**継承プロパティ**なので、
  画面の**ルート要素に `class="ui-root"` を付ける**だけで全子孫がSDF丸ゴ(`KokoSimFont`)に切替わり鮮鋭化する。
  画面ごとに root クラス名（`.tp-root` 等）が違ってもフォント指定を各所へ直書きせず、必ず `ui-root` を併記する。
  （深掘りが要る場合の FontAsset 側の SDFAA 構成・Retina等倍キャプチャ手順は別途「UI文字鮮鋭化」ノート参照。）

## ディレクトリ構成

```
Assets/UI/tokens.uss              # Step 1: 全トークン（唯一の正）
Assets/UI/Components/<Name>.uxml  # Step 2: 部品辞書（1部品=1ファイル）
Assets/UI/<Screen>.uxml + .uss    # Step 4: 画面（部品の組み合わせ）
Assets/KokoSim/<Feature>/*.cs     # ViewModel + Controller（エンジン出力の表示のみ）
screenshots/                      # Step 3: 自動出力の検証用PNG
```

## 運用ルール（要約）

1. モックは参照専用。数値と部品が正。
2. トークンにない色・サイズ・余白を直書きしない。
3. 新しい見た目は辞書に追加してから使う。
4. 部品はスクショ自動化で自己検証。人の目視は最終段のみ。
5. UIはエンジン出力の表示に徹する（ロジックを持ち込まない）。
