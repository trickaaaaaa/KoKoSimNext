## Q22. 夏場の気温連動オフェンス増加（乱打戦化）の実装方針（2026-07-22）→ **置き場所確定（2026-07-22・実装は別セッション）**

Issue: [#120](https://github.com/trickaaaaaa/KoKoSimNext/issues/120)

> **置き場所確定（2026-07-22・ユーザー確定）**: 天候（気温）は **大会単位の味付け（design-05）** に置く。
> 日程・地域・時間帯からシード付き専用ストリームをForkして**試合ごと**に気温を生成し、`GameContext` へ注入する
> （裏試合フルシム化 #43 の `GameContext` にも同じ注入口が通せる＝大会進行と同じ層に一本化）。カレンダー連動（design-03）は不採用。
> 残る実装前論点（変換式の可動域・帯追加の単独/一括・着手タイミング）は #120 で実装着手時に確定。決定コメント: [#120](https://github.com/trickaaaaaa/KoKoSimNext/issues/120#issuecomment-5045360671)。

**背景**: ユーザー提案「夏はもっと乱打戦になった方がいいのでは」から検討。「夏だから得点が入りやすい」を
確率へ直接乗せるのは不変条件#1（二層構造）違反になるため、物理層（気温→空気密度・投手消耗）で
効かせる方針とする。

**採用スコープ（2026-07-22・ユーザー確定）**: 案2「試合ごとの天候案（本格）」。日程・地域・時間帯に応じて
気温が試合ごとに振れる。決定論を保つため、天候はシード付き専用ストリームをFork生成する（不変条件#2）。
裏試合フルシム化（Q15。[#43](https://github.com/trickaaaaaa/KoKoSimNext/issues/43)＋
[#80](https://github.com/trickaaaaaa/KoKoSimNext/issues/80)＋
[#82](https://github.com/trickaaaaaa/KoKoSimNext/issues/82)の一括着手）とも絡むため、
着手順序はそちらの方針を踏まえて判断する。

**技術的な柱（2レバー、物理層のみ）**:
1. 気温↑→空気密度ρ↓→弾道の抗力減で打球が伸びる。`Aerodynamics.AirDensity`
   （`engine/KokoSim.Engine/Match/Pitching/Aerodynamics.cs:17`）は**既に存在する差し込み口**で、
   `data/coefficients.yaml` から一度だけ生成されるグローバル値（`CoefficientsFile.cs:174`）。
   試合ごとに変えるには、気温から `aero with { AirDensity = ... }` した派生インスタンスを
   試合単位で作る配線が要る（新規の口の追加ではなく既存レコードの試合単位差し替え）。
2. 暑さ→投手消耗（`PitchingFatigue`, `engine/KokoSim.Engine/Match/Game/PitchingFatigue.cs`）の
   `FatigueCoefficients`（`VelocityDropPerOverPitch`/`ControlDropPerOverPitch`）に気温項を加算し、
   終盤の球速天井・制球低下を早める。**野手側の疲労概念は2026-07-17に廃止済みのため復活させない**。

**要判断**:
- 天候（気温）という新概念の置き場所: カレンダー連動なら design-03、大会単位の味付けなら design-05
- 気温→AirDensity/消耗係数の具体的な変換式と範囲（現実の夏の甲子園も極端な乱打ではないため、
  控えめな係数から始める）
- `data/balance-targets.yaml` に夏場の得点/長打帯を新規追加し、統計回帰（Heavy）で確認。
  他の帯再校正と一括で入れるか単独で入れるか（統計系は溜めて一括の方針）
- 着手時期: 裏試合フルシム化（#43+#80+#82）等の優先保留作業との兼ね合い

## Q21. AI校（相手校）の Condition が常に Normal 固定（issue #48 実装時に発見・2026-07-22）→ **実装済み（2026-07-22・(a)当季Fork抽選）**

> **実装済み（2026-07-22）**: 決定=**(a) Fork抽選で付与**。`AiTeamBuilder.BuildFrom`（engine `Nation/Roster/`）で
> 相手校の各選手に、自校と同じ定常分布 `N(0, σ_stationary²)`（`FormModel.SampleInitialCondition`）から
> **(校ID, 年度, 選手ID) の決定論シード**で調子を抽選し `Player.Condition`/`ConditionValue` へ焼く。
> 展望（TournamentPreview）と実戦は同じ (校ID, 年度) でこの純関数を通るため観測は**当季を通して固定**＝
> 「展望で見た調子がそのまま出る」一致契約と決定論（不変条件#2）を保つ。これで #48 のオーダー編成・代打判断が
> 相手校でも実効化し、打席解決の `ApplyBatter`/`ApplyPitcher` にも調子が乗る。`AiRosterDeps.Form` を追加。
> 検証: 非Heavy 942緑・Nation 185緑・Heavy帯不変（AI校調子は StrengthTeamFactory 経路の帯テストを通らないため）。
> **残（将来増分）**: 試合ごと（ラウンド単位）に揺らす per-round variance は、展望との一致にラウンド識別子の
> `BuildOpponentTeam` 伝播（Unity Shell 配線）が要るため見送り。当季固定で十分な体験になっている。

issue #48（敵AIのオーダー編成・代打判断に調子を反映）を実装する過程で発見。実装自体は完了しているが、
**AI校（相手校）の試合では実際には一切効果を発揮しない**状態になっている。要判断ではなく実装済み機能の
限界の記録だが、次にこの効果を活かすには方針判断が要るためここに残す。

- **現状**: `StrengthTeamFactory.Create`（相手校の Player 生成）は `Condition`/`ConditionValue` を一切
  設定しない＝常に既定値（`Condition.Normal` / `ConditionValue = 0.0`）。週次の調子ランダムウォーク
  （`FormModel.NextWeeklyCondition`）は `SeasonEngine` が自校ロスター（`DevelopingPlayer`）にのみ適用しており、
  相手校は毎回その場生成（校ID＋年度からの決定論生成）のため持ち越す状態がない。
- **今回実装した範囲**: `StandardTacticsBrain.CallPinchHit`/`ComposeBattingOrder`・`AiTacticsBrain` 側は
  `Player.Condition` を正しく参照するので、Condition が変動する自校ロスターの選手（例えば自校の控え）を
  そのまま使うテストでは意図通り動く。`PlayerMatchResolver.BuildOpponentTeam` にも `ComposeBattingOrder`
  を配線済みだが、相手校の Condition が常に Normal のため**現状は恒等（並び替えなし）にしかならない**。
- **要判断**: 相手校にも試合ごとの調子variance を持たせるか。持たせる場合、(a) 校生成と同じ決定論契約
  （校ID＋年度のみに依存、観戦の有無で変わらない）を維持しつつ Fork ストリームで初期抽選するか、
  (b) 週次ランダムウォークに相当する何かを大会進行に連動させるか、(c) 現状維持（相手校は常にNormalのまま、
  この機能は自校の委任采配限定の効果と割り切る）。(a)/(b) は Q16（相手校の決定論生成契約）と同根の論点。

Issue: [#48](https://github.com/trickaaaaaa/KoKoSimNext/issues/48)（実装は完了・本項は追加の方針判断向け）

## Q18. デバッグモードの残未決（設計書17 起票時・2026-07-21）

Issue: [#85](https://github.com/trickaaaaaa/KoKoSimNext/issues/85)

設計書17（デバッグモード）は主要論点を §0 で確定済み。**Q18-1・Q18-2 とも確定（2026-07-21）＝本項クローズ**。

### Q18-1. 製品ビルドにデバッグモードを残すか → **確定: (b) 母種ログだけ残す（2026-07-21・ユーザー確定）**

`KOKOSIM_DEBUG` define で Development Build と Editor のみに限定する方針（design-17 §8）で書いたが、
選択肢は3つあった。

- (a) 完全に落とす: リリースに1バイトも入らない。安全。ただしプレイヤーからのバグ報告時に
  「再現トークン」を取れない（母種ログも出ない）
- **(b) 母種ログだけ残す** ← 採用
- (c) 隠しコマンドで解禁: リプレイ性・実績の意味が薄れるため不採用

**確定内容**:

1. **リリースビルドでも母種を1行ログに出す**: `[KokoSim] master seed = 0x....`。
   `GameSeed.InitMaster()` の直後、`KOKOSIM_DEBUG` の**外**（define非依存）に置く。
2. **出すのは観測だけ**。HUD・DebugBridge・シナリオ注入・強制発動は従来どおり `KOKOSIM_DEBUG` で型ごと消す。
   母種は「読み取り専用の16桁の数字」でありプレイヤーが操作に使える経路を作らない＝チートにならない。
3. **`GameSeed.Reset(ulong)` はリリースで公開しない**（`KOKOSIM_DEBUG` の内側に残す）。
   母種の**表示**とデバッグメニューからの**固定**は別物として扱う。
4. 出力先は Unity の `Debug.Log`（Player.log に残る）。プレイヤーに「ログを添えて報告してください」と言えば
   その1行で全体を再現できる状態にする。

**設計書17 §8 へ反映済み**（リリース混入防止の節）。

### Q18-2. デバッグシナリオYAMLの置き場 → **確定: (b) `data/debug/` 配下（2026-07-21・委任により決定）**

不変条件#4「バランス係数・球種定義・イベント・校名語彙・球場は `data/` のYAMLに置く」はゲームデータの規定であり、
デバッグ用シナリオはゲームデータではない。候補は (a) `data/debug-scenarios.yaml` 直置き /
**(b) `data/debug/scenarios.yaml`** / (c) `tools/debug-scenarios.yaml` の3つで、**(b) を採用**。

**確定内容**:

1. **パスは `data/debug/scenarios.yaml`**。既存のYAMLローダ・StreamingAssets 同期経路をそのまま流用できる
   （(c) は Unity の StreamingAssets 経路から外れ、`DebugBridge` 側にパス解決を別途書くことになるので却下）。
2. **`data/debug/` は「デバッグ専有ディレクトリ」として予約**する。将来の固定ロスター・再現ケース集などもここに置く。
   ゲームデータ（`data/*.yaml`）とデバッグ資材が同じ階層で混ざらないようにするのが (a) を採らない理由。
3. **リリースビルドから `data/debug/` をディレクトリ単位で除外**する。Q18-1 の「操作は消す」と同じ線で、
   注入用のシナリオ定義は製品に同梱しない。除外はファイル列挙ではなくディレクトリ1行で書けること自体が (b) の利点。
4. **`data/debug/` の欠損を正常系として扱う**。ディレクトリが無ければシナリオ0件で起動し、例外を投げない
   （リリースビルドでは常に欠損しているため）。

**設計書17 §3.4 / §8 へ反映済み**。

## Q20. 怪物（Phenom）の出現 — AI校の才能外れ値（2026-07-21）→ **実装済み（2026-07-22・#43/#80と一括）**

Issue: [#82](https://github.com/trickaaaaaa/KoKoSimNext/issues/82)

> **実装済み（2026-07-22）**: `Player.PhenomType`（隠しフラグ）＋ engine `Nation/Roster/PhenomPackages`。
> 新入生コホート生成時に全4000校一様でロール（尖り型5種＝主軸S帯＋支持底上げ／総合型）。逆算配分の対象外＝
> 上乗せ例外（怪物在籍校のチーム総合は Strength を上回る）。係数は `data/coefficients.yaml` の `persistent_roster.phenom_*`。

**背景**: 現行のAI校選手は各能力を Gaussian(学校Strength, σ=6) の独立ロールで生成するため
（`StrengthTeamFactory.Ability`）、中堅・弱小校に突出した個人（怪物）は構造的に出現しない。
設計書には怪物の概念があるのに生成側が未対応:
design-04 §2（新入生の外れ値・突出総合力／一芸特化）・design-05 §1.4（新聞の「怪物のうわさ」）・
design-02（S帯=世代の怪物）。ユーザー判断: **AI校にも怪物を出現させる**。

決定事項（2026-07-21・ユーザー確定）:

1. **Strength整合 → 上乗せ例外**。怪物は Q19 の逆算配分の対象外とし、チーム総合が学校Strengthを
   上回ることを許容する。勝てば Evolve 経由で Strength/Fame が追いつく＝「無名校の旋風」が創発。
   卒業（3年）と平均回帰で自然収束するため暴走しない。帯再計測は必須（Q19/#43の単独再計測に同乗）
2. **出現分布 → 全校4000校一様**。名声重み付けはしない。強豪の強さは既に Strength で表現済みのため、
   怪物枠は「弱小・中堅にこそ物語が生まれる」役割に専念させる
3. **タイプ2種**（design-04 準拠・比率と能力帯は係数化）【同日改訂: 尖り型をパッケージ化】:
   - **尖り型（パッケージ）**（主・期待 年1〜2人/全国）: 単発1能力のS化では「怪物感」が出ない
     （ユーザー指摘）ため、**主軸能力S帯(90+)＋同系統の支持能力を70〜80帯へ底上げ**する「型」で生成する。
     パッケージ定義（主軸＋支持能力の組と帯）はYAMLに列挙し、例:
     - 剛腕（投手）: 球速S＋キレ・スタミナ底上げ（制球は学校準拠＝ノーコン剛腕の怪物味を残す）
     - 超技巧（投手）: 制球S＋キレ底上げ
     - スラッガー（野手）: パワーS＋ミート底上げ・弾道高め
     - 韋駄天（野手）: 走力S（盗塁・走塁も連動）＋ミート底上げ（出塁してこそ怖い）
     - 鉄砲肩（捕手/外野）: 肩S＋守備・捕球底上げ
   - **総合型**（稀・期待 数年に1人/全国）: ポジション相応の主要能力を高帯へ＝S級（design-02 の
     「世代の怪物」）。こちらは従来どおり
4. **タイミング**: 新入生コホート生成時（Q19）にロール。セーブ開始時の3学年ブートストラップにも
   同確率で遡及適用（初年度の2・3年生にも存在し得る）。入学時から突出（＝うわさが立つ存在）。
   Q19 の節目成長には乗るが、成長予算の逆算配分からは除外（例外扱いを貫通）
5. **フラグ**: 選手に隠しフラグ **Phenom** を持たせ、新聞ダイジェスト（design-05・未実装）・
   大会展望の注目選手・テストから参照可能にする。用語集に「怪物 / Phenom」を追記
6. **自校との関係**: 自校スカウトの怪物（design-04 §2・talentCenter 未接続）は別トラックで本件は
   AI校のみ。将来スカウト接続時に出現率係数を共有できるよう係数名は自校/AI共通の置き場にする
7. **係数**: 出現率・タイプ比・能力帯はすべて `data/coefficients.yaml`（不変条件#4）

依存: [#80](https://github.com/trickaaaaaa/KoKoSimNext/issues/80)（永続ロスター＝新入生コホート生成が
実装の入れ場所）。#80 と同時実装が望ましい。

## Q19. 他校ロスターの3年永続化 — 選手個人の継続と節目成長（2026-07-21）→ **実装済み（2026-07-22・#43/#82と一括）**

Issue: [#80](https://github.com/trickaaaaaa/KoKoSimNext/issues/80)

> **実装済み（2026-07-22）**: engine `Nation/Roster/`（AiPlayer/AiSchoolRoster/AiRosterFactory/AiTeamBuilder/
> AiRosterCycle/NationRosters）。(校ID,入学年度)シードで学年コホートを決定論生成し、入学〜引退の3年在籍・
> 年3節目の逆算配分成長（翌夏チーム総合≈進化後Strength）。Shell（NationService/PlayerMatchResolver/GameClock）で
> 展望・実戦・成長駆動を配線。`StrengthTeamFactory.ForSchool` はブートストラップと集計モデル経路の互換用に存続。

**背景**: 現行のAI校は選手を持たず、`School.Strength`（学校単位スカラー）から対戦・展望のたびに
（校ID＋年度）シードで決定論生成する（`StrengthTeamFactory.ForSchool`）。年度が替わるとロスターは
丸ごと別人になり、「秋に対戦した2年生エースが翌夏に成長して戻ってくる」体験が存在しない。
ユーザー判断: **毎年の使い捨て生成は不可。選手個人が入学〜引退の3年間在籍し続けること**。

決定事項（2026-07-21・ユーザー確定）:

1. **主従関係 → (a) Strength主・成長は逆算配分**。`AiSchoolModel.Evolve`（年次進化＝平均回帰＋世代交代
   揺らぎ＋大会成績＋名声。30年生態系は帯検証済み）は現行維持。代替わりサイクルの1周
   （夏後Evolve → 秋新チーム → 4月新入生加入 → 翌夏）で、**翌夏時点のチーム総合が進化後Strengthに
   一致する**よう、在校生の年間成長予算と新入生の質を決定論で解く（逆算配分）。
   選手から見れば自然な成長カーブ、学校から見れば従来どおりの生態系
2. **成長の節目 → 年3節目**（夏大会後=代替わり・秋大会後・冬明け）。自校の週次成長とは別系の
   AI校簡易成長で、年間予算を3分割して適用。8万人超の週次処理はコストと意味が釣り合わないため不採用
3. **ライフサイクル**: 4月に新入生コホート加入（質はStrength/Fame由来・(校ID, 入学年度)シードで
   決定論生成）。夏大会敗退で3年生引退（新チーム移行）。背番号・打順・守備位置は代替わり時に再編成
4. **スコープ**: 全4000校。在籍は各学年約10人・計約30人を保持し、ベンチ入り20人はTeam構成時に選抜
   （3年生引退直後の秋でも20人が組めることを保証）。人数は係数化して調整可能にする
5. **決定論契約の維持**: ロスター状態は「カレンダー節目の純関数」＝観測・対戦の有無で変わらない
   （Q16(a)「状態のみ注入」契約と整合、不変条件#2）。`StrengthTeamFactory.ForSchool` の都度生成は
   セーブ開始時の3学年ブートストラップ専用に役割変更
6. **Q15との関係**: 未決1の決定「(a)大会開始時に一括実体化（年間育成はティア抽象のまま）」を
   **「永続ロスターから構成」に上書き**。[#43](https://github.com/trickaaaaaa/KoKoSimNext/issues/43)
   実装時にこの前提を織り込む（使い捨て実体化を作ってから置換する二度手間を避ける）
7. **帯への影響**: 相手校能力の季節分布・年またぎ相関が変わるため帯再計測は必須。
   Q15未決4の決定（#43単独で帯再計測）に同乗させ、一括バッチには載せない

## Q18. コールドゲームが実ゲームフローで未適用（大会単位トグルの設計判断・2026-07-21）→ **方針確定（同日・5論点確定）**

エンジンには MercyRule（5回以降10点差 or 7回以降7点差で打ち切り。`GameEngine.IsMercy`,
engine/KokoSim.Engine/Match/Game/GameEngine.cs:962）が実装済みだが、スイッチの
`GameContext.MercyRuleEnabled`（GameContext.cs:36）はデフォルト false のまま、実ゲームフローの

## Q14. 打球のバウンド表現が存在しない（高バウンド内野安打・バウンドで内野を越える打球・イレギュラー）（2026-07-20）→ **方針確定・Issue化（2026-07-21）**

> 決定: (a) **物理層のバウンド積分**（反発係数・摩擦で着地後も継続、迎撃点は時間×高さ）。
> (b) **4分類へ拡張**（ゴロ/バウンド/ライナー/フライ、Playback/2D俯瞰へ波及込み）。
> (c) **イレギュラーは一律小確率で今回同時に入れる**（球場の土質/荒れ接続は将来の別Issue）。
> Issue: [#63](https://github.com/trickaaaaaa/KoKoSimNext/issues/63)。帯再校正の一括バッチ対象（残は本件のみ）。

現行の守備解決（`FieldingResolver.ResolveDetailed`）は弾道積分を**最初の着地点で打ち切り**、着地後の
跳ね方・転がりを一切シミュレートしていない。打球は「最初の着地点の距離＋滞空時間＋頂点高さ」だけで
フライ/ゴロに二分され、以下の現実のプレーが構造的に表現できない（設計書12/14にも項目なし）。

- **表現できないもの**:
  - (1) **高いバウンド（チョッパー/叩きつけ）で野手が待たされる間に生きる内野安打**。頂点が
    `FlyApexThresholdM` 以上の打球はフライに分類され「高く弾んだゴロ」という状態が存在しない。
  - (2) **ゴロがバウンドで内野手の頭上/グラブを越えて外野へ抜ける打球**。着地点が `InfieldDepthM` 以内なら
    必ず最寄り内野手の処理対象になり、抜けた場合も「タイム負けの内野安打（Single）」として畳まれる。
    外野へ転がって走者が余分に進む二次展開はない。
  - (3) **イレギュラーバウンド起因の失策/安打**。エラーは捕球能力ベースの一律確率（`MaybeError`）のみ。
- **表現できているもの（誤解防止）**: 足で稼ぐ内野安打（守備時間 vs 一塁到達のタイム勝負）と、
  空中で内野の頭を越えるポテンヒット（フライ判定で誰も間に合わない落下点）は既に創発する。
- **要判断**:
  - (a) モデルの深さ: 物理層に忠実なバウンド積分（反発係数・摩擦で着地後の軌道を継続し、内野手の
    迎撃点を「時間×高さ」で判定）まで踏み込むか、簡略層（打球の垂直角/頂点高さから「バウンド滞空時間」を
    近似して守備時間に加算する程度）で止めるか。不変条件#1（物理層解決）に沿うなら前者が筋。
  - (b) 分類の再設計: 現在の「フライ/ゴロ」二分（`FlyApexThresholdM`）を「ゴロ/バウンド/ライナー/フライ」へ
    拡張するか。タイムライン/2D俯瞰再生（PlaybackModel）への波及範囲も要確認。
  - (c) イレギュラーバウンドを入れるか。入れるなら球場（土質/荒れ）との接続（design-13 拡張）の要否。
- **帯への影響**: BABIP・内野安打率・エラー率・得点環境が動く**解決ロジックの変更**＝Heavy帯再校正＋
  決定論ベースライン再生成が必要。「帯再校正は溜めて一括」方針に沿い、フォース進塁修正（下記2026-07-19
  項）・実戦成長ループ（Q8）等と同時に回すのが効率的。

**前**（打席の投球ループに入る前）で呼んでいる。一方、打席途中で3アウト目になって打席が未決着のまま
終わるケース（スクイズの挟殺＝設計書15 Phase D-2c の `squeezeAbandoned`、および今回移植した盗塁死・
牽制死＝`stealEndedHalf`）は、いずれも `NextBatter()` の**後**（投球ループの中）で `outs>=3` を検知して

- コード中のコメント（例: `res.Squeeze` 処理直前, `squeezeAbandoned`/`stealEndedHalf` 分岐）は
  「3アウト目ならこの打席は未決着のまま次イニングへ（＝この打者は次イニング先頭）」と書かれており、

  この記述は一致する。しかし上記の理由で、コードの実際の挙動はコメントの説明と食い違っている
  （次イニング先頭になるのは1人ズレた次の打者）。
- 昔からある「盗塁死で3アウト」の経路（打席頭でPA開始前に解決していた旧実装）は `NextBatter()` より
  前に解決していたため、この経路では正しく「同じ打者が次イニング先頭」になっていた。今回、盗塁を
  毎球ループへ統一したことで、D-2c のスクイズと同じタイミング（`NextBatter()` の後）に揃ったため、
  この不一致は3経路（スクイズ・盗塁死・牽制死）すべてに共通する状態になった。
- **今回の対応**: D-2c で既に採用済みの挙動（`NextBatter()` 後に打席未決着で終わる）に盗塁死・牽制死も

  （D-2c時点の既存挙動を尊重し、スコープ外の判断を勝手にしないため）。
- **要判断**: (a) 実際の挙動（1人ズレる）を仕様として受け入れ、コメントの文言を実態に合わせて修正するか、
  (b) `NextBatter()` を投球ループの直前ではなく「打席が実際に確定した時点」まで遅らせて、コメント通りの
  野球ルールに挙動を合わせるか。後者は影響範囲がバッター識別（成績記録・スタメンハイライト等、
  `batter`/`batterOrder` を使う箇所）に及ぶため、単独の帯移動は無さそうだが確認が要る規模の変更。

## 2026-07-19 フォース走者の進塁が確率化されている（走塁モデルの帯再校正案件）→ **クローズ（2026-07-20・帯再校正バッチで修正）**

> 実装: `BaserunningModel.ApplyInPlayOut` にフォース判定を導入。ゴロ凡打（`home.IsFly==false`）では
> 打席開始時の占有から一塁走者=常時／二塁走者=一塁走者あり／三塁走者=満塁をフォースとし**確定進塁**
> （`ProductiveOutAdvanceProb` は非フォース専用に限定）。満塁×内野前進のみフォースの三塁走者は自重不可
> ＝送り判定なしの本塁封殺レース。フライ（タッグアップ）と home 無し経路は完全不変。
> 検証: `ForceAdvanceTests`（6件。1死1塁ゴロの一塁残留が構造的に不可能なことを固定）。
> 帯: 決定論ベースライン再生成（200カード中199変化）のうえ **Heavy 21/21緑＝`balance-targets.yaml`
> 再校正は不要だった**（得点押上げは既存帯内。10000試合レポート out/report-2026-07-20-force-hbp.md）。

MatchLive の2D俯瞰観戦で「1死1塁の内野ゴロで一塁走者が一塁ベースから動かない」という不自然さが報告された（2倍速で観測）。調査の結果、**表示バグではなくエンジンの走塁モデル `BaserunningModel.ApplyInPlayOut` の不正確さ**と判明。表示（`TimelineBuilder.AppendHeldRunners` の静止走者トークン）はエンジンの真値を忠実に描いているだけで、表示側だけでは正しく直せない（表示だけ二塁へ動かすと次打席でエンジン真値=一塁へワープする）。

- **原因**: [BaserunningModel.cs:396](engine/KokoSim.Engine/Match/Game/BaserunningModel.cs#L396) で、内野ゴロ（InPlayOut・併殺/野選なし）における一塁走者→二塁の進塁が `ProductiveOutAdvanceProb`（既定 **0.18**、[coefficients.yaml:98](data/coefficients.yaml#L98)）の確率判定になっている。この確率は同ファイルの二塁走者→三塁（**非フォース**の進塁打）と共用。しかし打者がアウト（一塁封殺）になった局面の一塁走者は**フォース（強制進塁）**なので、送球は一塁へ行った＝二塁は空き＝走者は二塁へ悠々セーフ、が野球的に正しい。18%でしか進まず**82%は一塁に取り残される**のはモデルの不正確さ。
- **正しい修正の方向**: フォース走者（打者アウトで押し出される一塁走者、および連鎖的に押し出される二塁・三塁走者）は**確定で進塁**させる。`ProductiveOutAdvanceProb` は非フォースの進塁打（二塁→三塁の右方向ゴロ等）専用に限定する。
- **なぜ今やらないか（要判断で見送り）**: これは**解決ロジック（resolution）の変更**で、走者が塁上・得点圏に多く残る→得点環境が上がるため **不変条件#2（決定論ベースライン再生成）・#5（Heavy帯の測り直し＋`balance-targets.yaml` 再校正）** が発生する。本チャットの一連の修正は「演出層のみ・解決/バランス不可侵」を貫いており、この境界を越える。メモリ「帯再校正は溜めて一括」方針に沿い、**後日まとめて対応**とユーザー判断（2026-07-19）。
- **対応時の手順メモ**: ①フォース判定（各塁が後方の占有で押し出されるか）を導入 → ②フォース走者は確定進塁・非フォースのみ `ProductiveOutAdvanceProb` → ③`EngineDeterminismGate` ベースライン再生成（`DeterminismBaselineDump` Skip解除）→ ④`GameRegressionTests`/Heavy 帯を測り直し、`data/balance-targets.yaml` を再校正（得点・残塁・併殺率の帯が動く見込み）。他の帯再校正案件（実戦成長ループ Q8 等）と同時に回すのが効率的。
