# 設計書06: UI設計（現代的マネジメントUI）

方針決定: 元祖風のテキスト主体ではなく、**現代的なマネジメントゲーム風ダッシュボード**
（Football Manager系の情報設計を高校野球の題材に翻訳する）

---

## 1. 実装技術: Unity UI Toolkit を採用

- 管理画面系はすべて **UI Toolkit（UXML/USS）** で構築する
- 理由: レイアウトとスタイルがテキストファイル（UXML/USS）で完結するため、
  **Claude Codeがエディタを介さずUIを直接書き・修正できる**。表・リスト中心の画面との相性も良い
- uGUI/3D空間は将来の試合3Dビュー専用に温存
- 構造: 純C#エンジン → ViewModel層 → UXMLバインディング（エンジンにUI依存を持ち込まない）

## 2. 情報設計の原則

- **ハブ＆スポーク**: ホームダッシュボードを中心に全機能へ1クリック
- **通知フィード駆動**: 週送りで起きたこと（イベント・新聞・成長・怪我）はすべてフィードに流れる。
  「まとめて進行」が止まった理由が常に一目で分かる
- **等級の視覚言語統一**: S〜Gに固定カラー＋文字併記（色覚対応）。能力・球種・ティア全てで共通
- PC（Windows）前提: 16:9、マウス主体＋キーボードショートカット

## 3. 画面一覧

### 3.1 ホーム（週次ダッシュボード）
今週のカレンダー / 次の試合カード / 部の状態サマリ（怪我人）/
通知フィード / クイックアクション（進行・練習計画・個別指導）

### 3.2 選手
- 一覧: ソート・フィルタ付きテーブル（守備位置・学年・能力・調子）。比較ピン留め機能
  （調子はパワプロ式アイコン↑↑〜↓↓で一目表示。設計書02 §3.3）
- 詳細: 能力チャート、成長推移グラフ、スキル、性格・見極め進捗（？→開示）、
  投手はStatcast風データタブ（球種ごとの変化量cm・回転数rpm ← 設計書02の内部2軸を見せる場所）

### 3.2b 成績（眺めてニヤニヤする中核画面）
1球単位エンジンの副産物として詳細成績が自動集計される。**3つの時間軸**で切り替えて見られる:
- **今大会成績**: 進行中の大会での暴れっぷり（リアルタイムに伸びる）
- **公式戦通算**: 高校3年間の対外試合の積み上げ（そのチームでの物語）
- **キャリア通算**: 卒業後も継続。教え子がプロ入り後も追える（設計書04 教え子名鑑と接続）

指標（物理エンジンがあるので現代的に）:
- **野手**: 打率・本塁打・打点・盗塁の基本＋出塁率・長打率・OPS、
  打球データ由来の平均打球初速/角度・最長本塁打距離
- **投手**: 防御率・**投球回**・奪三振＋K/9・WHIP・被打率、平均球速・最速の推移
  （高校野球なので勝敗は用いない。投球回で連投・酷使を可視化）
- 年度別/対戦相手別の内訳、月間MVP級のハイライト、キャリアハイの自動ピックアップ
- **記録室（設計書 3.9）連携**: 校歴代記録・全国記録に絡むと通知フィードで祝う

### 3.3 練習計画
週テンプレエディタ（ドラッグ割当）/ 強度設定 / 個別指導3枠の指名 / 委任トグル / テンプレ保存・呼出

### 3.3b 背番号・ベンチ入り登録
- 部員リストから**ベンチ入り20名を選び、背番号1〜20を割り当てる**ドラッグ操作が主体
- 選ばれなかった部員は「ベンチ外」ゾーンに残り一目で分かる（部員20名超の学校ではここが実質的なメンバー選考）
- 慣例番号の自動割り当て補助: 1=エース / 2=正捕手 / 3〜9=各守備位置 / 10〜=控え（強制ではなく提案）
- **大会ごとに登録し直せる**（夏用・秋用でメンバーが変わる。設計書03の年間カレンダーと連動）
- 番号1を誰に託すか等はドラマ要素。信頼度・士気に軽く波及させてもよい（要調整）
- 自動編成の委任にも対応（監督は最終確認だけ、も可）

### 3.4 試合
- 高速モード: スコアボード＋テキスト速報＋勝率インジケータ
- 詳細モード: **元祖踏襲の2ビュー構成**
  - 投打対決ビュー: 1球指示UI（ストライクゾーン図で配球・狙い球、作戦パネル）＋
    対決カットイン（デフォルメ静止画の紙芝居＋球速・球種・判定表示）
  - 打球が出た瞬間に**守備俯瞰ビュー**へカット（2D再生モックの本実装。カット時刻＝タイムラインの接触イベント）
- 同一タイムラインを2つのビューが描き分けるだけなので、エンジン側の追加実装は不要
- 画面中央は将来3Dビューに差し替え可能なコンテナとして設計

### 3.5 大会・全国
トーナメント表（自校ハイライト）/ 地方大会新聞 / 全国ランキング / 勢力図マップ（都道府県色分け）

### 3.5b 大会プレビュー（元祖の大会展望を踏襲・自動生成）
抽選後〜開幕前に自動生成。開幕前の高揚感を演出する“読み物”。材料は既存データから全て賄える。
- **大会展望**: 優勝候補◎ / 対抗○ / ダークホース▲ の格付け（各校総合力＝設計書05のAI校運営値から算出）＋概観文
- **注目選手ピックアップ**: 投手部門・打者部門。能力・成績・伸びしろの閾値超えを自動抽出
  （例: 最速150超の剛腕、通算本塁打上位のスラッガー）。各選手に寸評
- **出場全校の登録メンバー閲覧**: 各校の背番号入りベンチ入り20名を閲覧（§3.3bの登録データ）。
  表示は**背番号・名前・学年・投打（右投左打など）まで**に留め、詳細指標は選手詳細（§3.2b）へ誘導
  （展望段階で全校の細かい数字まで出すと情報過多になるため）
- **チーム寸評**: 能力バランスから特徴を文章テンプレに流し込み自動生成
  （例:「強力打線が武器だが投手層に不安」）。寸評テンプレはデータ駆動でClaude Codeが増やせる
- 地方大会新聞（§3.5）と連動。自校が候補に挙がると通知フィードで煽る

### 3.6 スカウト・新入生
入部予定者リスト（？表示→開示演出）/ 中学生リスト（勧誘対象）/ スカウト契約管理

### 3.7 学校経営
資金収支 / 施設・備品購入 / 合宿計画

### 3.8 監督キャリア
分野別能力 / 名声・信頼度 / キャリア年表（歴任校・戦績）/ 実績・称号 / 教え子名鑑（プロ入りOB）

### 3.9 記録・設定
歴代大会結果 / 記録室 / 学校エディタ / エクスポート対戦 / 現代ルールトグル

## 6. 共通トップバー規約（全画面共通・この先の決まり事）

画面上部の「スコアボード＋ナビ」は **`Assets/UI/Components/TopBar.uxml` を単一ソース**とする。
各画面で独自ヘッダーを作らない（過去に選手一覧が独自コンパクトヘッダーを持ち、上部UIが画面間で不一致になった。統一済み）。

**新しい画面を追加するときの決まり事:**
1. UXML 冒頭で `<ui:Template name="TopBar" src="Components/TopBar.uxml" />` を宣言し、ルート直下（**ScrollView の外＝固定トップ**）に `<ui:Instance template="TopBar" />` を置く。本文は別の ScrollView に入れる。
2. ナビのクリック導線とアクティブタブ強調は **`ScreenRouter` が自動配線**する（画面側で書かない）。ナビ項目名は `nav-home / nav-players / nav-training / nav-tournament / nav-scout / nav-management / nav-career / nav-records`。
3. スコアボードの動的値は各画面のコントローラが `name` で埋める:
   - `week`（現在日付）= **`GameClock.CurrentLabel()`** を必ず通す（画面ごとに書式を作らない）。表記は **「YYYY年M月W週目」**（年度1=2026年度・4月始まり。1〜3月は暦年+1）。週→暦の変換はエンジン `SeasonCalendar.DateOf`、暦年の起点は `SeasonClock` が正。
   - `team-rank` = 総合力ランクの等級チップ（`grade` + `grade--X`）。
   - `funds / fame / trust / countdown` 等はデータが無ければ TopBar 既定値のまま可（現状ホーム/練習も既定値依存）。
4. **現在週は全画面共有の `GameClock`（Shell層・セッション単一ソース）が持つ**。各画面の State は自前で週を持たず GameClock から読む。「今週を進める」(`advance`) はどの画面から押しても `GameClock.Advance()` で共有週を進め、**全画面の日付・合宿バナー・成長段階が連動**する。既定はシーズン頭（週0＝2026年4月1週目）で、Play開始（ドメインリロード）で週0へ戻る。
   ※ 部員データ等は依然として画面ごとに ephemeral（プロトタイプ）。共有するのは「現在週」だけ。
5. **一覧→詳細**の従属画面（選手詳細・試合・大会）は「戻る」動線が要るため、TopBar と戻るボタンの併存方式を後で確定する（**未決**）。現状これらは旧ヘッダーのまま。統一時はこの規約に寄せる。
