# 設計書11: 敵AI（試合采配の意思決定）

対戦相手の采配ロジック。**プレイヤーとAIは同じ采配システム（設計書09）を使い**、
AIは「どの選択肢をどう選ぶか」の意思決定だけが異なる。AI専用の裏ルールは作らない
（＝AIだけ有利/不利にしない）。強さと個性を三層で作る。

## 0. 三層構造

| 層 | 担うもの | 由来 |
|---|---|---|
| **① 能力値**（ミス率・ブレ） | 判断の正確さ。弱い監督はミスする | 設計書04 AI校の采配能力 |
| **② 段階ロジック**（戦術の引き出し） | そもそも使える戦術の高度さ | AIティア（下記2） |
| **③ 校風**（傾向の重み） | 采配の好み・チームの色 | 学校属性（設計書05） |

- ①で「正しく打てるか」、②で「高度な手を知っているか」、③で「どんな野球を好むか」が決まる
- 三層が掛かり合い、対戦相手が「強さ×高度さ×性格」でばらける

## 1. 能力値層（ミスとブレ）

AI校の采配能力（設計書04）を、判断の質に反映する。弱い監督ほど:
- 継投判断が遅れる（エースが崩れても引っ張る／逆に早すぎる）
- サイン・作戦の選択を誤る（場面に合わない強攻・無謀な盗塁）
- 相手（プレイヤー）の作戦察知に失敗する（バント/スクイズを見抜けない）
- 伝令の使いどころを外す（動揺を放置、無駄打ち）
- 守備陣形・配球方針の設定が最適から外れる

実装: 各判断に「最適解を選ぶ確率」を采配能力で変動。低いほど次善・悪手を引きやすい。
完全ランダムではなく「正着からのブレ」として表現（弱くても時々good playはする）。

## 2. 段階ロジック層（AIティア）

学校の強さティア（設計書05）に応じて、使える戦術の引き出しを段階化する。

| ティア | 使える戦術の範囲 |
|---|---|
| 初級（G〜F） | 基本のみ。バント・単純な継投・代打程度。配球は素直 |
| 中級（E〜C） | カウント別配球、状況に応じたバント/盗塁、左右の代打、守備位置調整 |
| 上級（B〜A） | 配球の緩急と釣り球、守備シフト、継投の先読み、相手の弱点を突く攻め、伝令の的確な運用 |
| 名門（S） | 上級＋こちらの作戦を高精度で察知し対抗（ウエスト・ピッチアウト・バスター見抜き等） |

- ティアは①能力値と独立: 「引き出しは多いが監督が凡将でミスる」中堅私立、
  「引き出しは少ないが堅実にこなす」公立の名将、といった掛け合わせが生まれる

## 3. 校風層（采配の個性）

学校属性（設計書05）に「采配傾向」を持たせ、選択の重みを偏らせる。同じ局面でも校風で手が変わる。

| 校風タイプ | 傾向 |
|---|---|
| 機動力野球 | 盗塁・バント・エンドランを多用。1点を積極的に取りに行く |
| 強打・待球 | 待球して長打狙い。バント少なめ、初回から大量点を狙う |
| 守り勝つ野球 | 継投・守備固めを厚く、僅差を守り切る。手堅い送りバント |
| 全員野球 | 代打・代走・継投を小刻みに動かす。選手層を活かす |
| 豪腕依存 | エースに徹底的に託す（継投が遅い＝諸刃）。打線は水物 |
| 型なし（一般校） | 突出した傾向なし。ティア相応の標準的な采配 |

- 校風は生成時に学校へ付与（設立・伝統・地域から確率的に。エディタで変更可）
- 校風は大会プレビュー（設計書06 §3.5b）の寸評材料にもなる（「機動力が武器」等）

## 4. 具体的な意思決定ポイント（プレイヤーと同じ選択肢を使う）

| 局面 | AIの判断入力 |
|---|---|
| オーダー編成 | 能力＋校風（俊足を上位／強打を中軸）＋左右のバランス＋調子＋怪我状態 |
| 継投 | 投手のスタミナ（設計書02）＋失点状況＋残り試合（大会日程）＋校風（豪腕依存は遅い） |
| 代打 | 対左右の相性（慣れ）＋調子＋残り選手層＋ティア（上級ほど的確） |
| 攻撃サイン | 点差・回・走者・校風の重み＋能力（バント/盗塁パラメータ）＋相手投手 |
| 守備陣形・配球 | 打者の傾向＋点差＋ティア（上級はシフト/釣り球を使う） |
| 伝令 | 投手の動揺・プレッシャー状況＋残回数管理（上級ほど温存が上手い） |
| 主将起用 | 校風・ティアで「実力優先」か「支柱をあえて使う」かが変わる |

- すべて設計書09の采配システムの選択肢を選ぶだけ。新しい行動は追加しない

## 5. 裏試合との整合

- 自校の試合: 上記フルロジックでAIが采配
- 中解像度・集計モデルの裏試合（設計書05）: フルロジックは回さず、
  三層（能力値・ティア・校風）から勝率・スコア分布を直接サンプリング。
  フル采配の統計結果にキャリブレーションして整合を保つ

## 6. 実装・検証

- 意思決定は data 駆動の重みテーブル＋ルールで実装（校風・ティアの係数はYAML）
- 検証（設計書07）: AIティア別の勝率、校風別の戦術発生率（機動力校の盗塁企図数等）が
  意図通りか集計。プレイヤー不在の「AI vs AI」を大量試行してバランス確認
- 難易度: プレイヤー体験としての難易度は、対戦相手のティア分布（＝どのレベルの学校と当たるか）で
  自然に決まる。別途の難易度補正は持たせない（現実の高校野球の非対称性で十分）
