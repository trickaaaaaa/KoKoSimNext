## 0. 方針（3層に分け、①のみ今実装）

| 層 | 内容 | 扱い |
|---|---|---|
| ① 物理（広さ） | 両翼・中堅距離、フェンス高 | **今回実装** |
| ② 環境 | 球場固有の風（浜風等）、屋根 | 後回し（風雨システムとセットで将来判断。データの箱だけ用意） |
| ③ ビジュアル | スタンド・土の色などの見た目 | 後回し（3D/俯瞰の描画テーマ。フェンス弧の半径だけ①から反映） |

理由: エンジンは打球を初速・角度から物理計算しているため、**フェンス距離・高さをパラメータ化する
だけで狭い/広いが試合結果に自動で効く**。新しい仕組みは不要でコスパが突出して高い。

## 1. 球場データ（YAML・データ駆動）

data/stadiums.yaml（新規）:

```yaml
stadiums:
  - id: koshien
    name: 甲子園球場        # 聖地のみ実名想定（他は架空名）
    left: 95                # 両翼 [m]
    center: 118             # 中堅 [m]
    fence_height: 2.6      # フェンス高 [m]
    tier: national          # national / prefectural / municipal
    # wind: { ... }         # ②用の箱。今回は未使用
  - id: pref_large_01
    name: （県営級・広い）
    left: 100
    center: 122
    fence_height: 3.0
    tier: prefectural
  - id: municipal_small_01
    name: （市営級・狭い）
    left: 91
    center: 115
    fence_height: 2.0
    tier: municipal
```

- 寸法は実在準拠のばらつき: 両翼91〜101m / 中堅115〜122m / フェンス高2〜4m
- 架空球場を数タイプ（広い県営級 / 標準 / 狭い市営級）用意し、県に2〜3球場を割当
- 甲子園は**両翼狭め（95m）＋中堅深め（118m）の変則球場**（実寸準拠。stadiums-research.md 実測）。
  地方決勝クラスの多数派（両翼98〜101m/中堅122m）より両翼が浅く、**両翼への一発はむしろ出やすい**
  （40000打席実測: 甲子園HR% 3.22% ＞ 地方広め2.03%）。「聖地の格」は寸法でなく
  **プレッシャー指数・観客・演出**の側で表現する（※旧記述「広め＋高フェンスで本塁打が出にくい聖地」は
  実寸と真逆だったため2026-07-20に改訂。Q11クローズ）

## 2. エンジン接続（①の実装）

1. **打球のフェンス判定を球場参照に**: 本塁打判定・フェンス直撃（エンチタイトル二塁打含む）・
   外野の守備範囲の背走限界を、固定値でなく試合の球場データ（方向別のフェンス距離を左中右で補間）から引く
2. **大会→球場の紐づけ**: 大会データ（pref-formats）に使用球場リストを持たせ、試合ごとに割当
   （序盤ラウンド=市営級中心、準決勝・決勝=県営級、甲子園=koshien固定）
3. **2D俯瞰ビュー**: フェンス弧の描画半径を球場データから取得（狭い球場は見た目にも狭い）
4. **タイムライン出力は不変**: 球場はエンジン内部の判定パラメータであり、出力契約（CHANGELOG 32）は変わらない

## 3. ゲーム的な意味（狙い）

- **采配の変数**: 「今日は狭い市営球場、非力な打線でも一発が狙える」「決勝は広い県営、外野の脚と肩が問われる」
  — 抽選で球場が決まるたびチーム特性との相性を読む楽しみ
- **チーム作りへの波及**: 狭い球場中心の県では一発長打型が、広い球場では走力・外野守備型が相対的に得をする
- 大会プレビュー（設計書06 §3.5b）の寸評材料にも使える（「決勝の舞台は広いXX球場。守備力が問われる」）

## 4. 検証（設計書07の流儀）

- 同一チーム同士を「狭い球場」「標準」「広い球場」で各1万試合: HR%・長打率・平均得点が
  狭>標準>広 の順に並ぶこと、差が現実的な幅（HR%で±1%程度）に収まることを確認
- 甲子園設定でPhase1指標を再出力し、**地方の広い球場より本塁打がやや出やすい**こと
  （両翼95mの変則寸法の帰結。実測: 甲子園3.22% ＞ niigata 100/122 2.03%）を確認
  （※旧記述「出にくいことを確認」は実寸と矛盾したため2026-07-20改訂。統計回帰テストは
  「架空3タイプ 狭>標準>広」の順序＋「甲子園>地方広め」を検証する形で実装済み）

## 6. 球場ビジュアルの3段階（③の方針確定・実装はビジュアル着手時）

デザイン投資を「プレイヤーが体験する頻度と感情の重み」に比例させる3段階構成。

| Tier | 対象 | デザイン | コスト戦略 |
|---|---|---|---|
| 1 | 架空球場（序盤ラウンドの市営級） | 汎用デザイン1式 | フェンス距離・色味のパラメータ差し替えで無限に使い回し |
| 2 | リサーチ済み47球場（決勝クラス） | リッチ共通テンプレート1式 | 寸法（フェンス弧の形）・名前表示・カラーリングで県ごとに差別化。個別制作なし |
| 3 | 甲子園・神宮 | 固有デザイン（専用アセット） | 聖地2つのみ手をかける（甲子園=蔦・銀傘・黒土 / 神宮=都会の夜空） |

- **演出の階段がゲーム進行と一致**: 序盤=汎用 → 決勝=リッチ化で「勝ち上がった」を画面が語る →
  甲子園=別格の画面。豪華さ自体が到達度の演出装置になる
- Tier 2に**地域色バリエーション軸**を1つ持たせる: スタンド色調・背景（山並み/海/都市）を県データから
  2〜3パターン選択。パレット差し替え程度のコストで「土地ごとの風景」を出す（転戦のロマン）
- 2D俯瞰はタイムライン再生の背景レイヤー差し替えなので、試合エンジンには無影響

### 6.1 アセット調達方針（提案）

前提: 個人開発・Claude Code主体。「描く」より「生成・調達して整える」を基本線に。

1. **2Dの間は自前生成で足りる（推奨・まず これ）**:
   - 俯瞰ビューの球場は幾何描画（フェンス弧・芝・土）＝**コードで描く**。既にモックで実証済み。
     Tier差は配色・スタンド帯の装飾・観客ドットの密度で表現でき、アセット制作ゼロ
   - 背景の地域色もグラデーション＋シルエット（山/海/ビル群）の程度ならコード描画で成立
2. **一枚絵（試合前のスタジアム外観カット等）はAI画像生成**:
   - Tier 3の甲子園・神宮の「入場時の一枚絵」だけAI生成＋手直し。商用可のツール
     （Midjourney/DALL·E等の商用ライセンス）で生成し、実在球場に似せすぎない様式化
     （権利・肖像の安全域。蔦の球場"風"、都会の夜のスタンド"風"に留める）
   - Tier 2用は共通テンプレ1枚＋色替えで済ませる
3. **3Dに進んだ場合**:
   - ベースはアセットストア（Unity Asset Store等）の汎用スタジアムを1つ購入し、
     マテリアル・色・フェンス寸法をパラメトリックに差し替え（設計書08のパラメトリックユニフォームと同じ思想）
   - Tier 3のみ装飾モデル（蔦テクスチャ・銀傘形状）を追加制作
4. **やらないこと**: 47球場の個別モデリング/個別イラスト発注。写真そのままの使用（権利・解像度の沼）

- 結論: **2Dフェーズはアセット調達ゼロ（全部コード描画）で開始できる**。一枚絵とTier 3装飾だけが
  将来のアセット作業で、それもAI生成＋既製アセットで賄う。個人開発の物量に収まる
