## 1. 基本方針（確定・再掲）

**リアル頭身（6〜7頭身）のローポリ3Dモデル＋ピクセル化シェーダ。**

- モデルはリアル、画面に出る絵はドット風（解像度量子化・色数制限・輪郭）
- 3Dのためキャラ一貫性は構造的に保証。全ポーズ・全アングル・左右ミラー・色替え自由
- 顔なし方針（設計書08）と両立: ピクセル化で顔が数ピクセルに潰れ「描かれていない」ことが自然になる
  - リアル頭身の生ポリでのっぺらぼうは不気味の谷に落ちるため、シェーダ必須
- 俯瞰ビューの丸トークンは置き換えない。3Dキャラの出番は対決ビューのみ
- 将来の3D俯瞰観戦（α後スパイク・非本線）への滑走路を兼ねる

### 1.1 シェーダが調達要件に与える帰結（重要）

ピクセル化を通すと、テクスチャのディテール・PBR質感・顔の造形は全て潰れる。
モデルに要求されるのは **「6〜7頭身のシルエット」と「フラットな配色」のみ**。

- 高品質リアル人体（スキャン系等）は完全にオーバースペック → 調達不要
- フラットカラーのローポリ素体が最適解
- 調達コストの重心はモデルではなく **モーション側** に置く

### 3.2 素体モデル

- **スパイク用（確定）**: 無料CC0ローポリ人型（Quaternius系等）。リグ付き・Humanoid化容易
  - ややスタイライズだがピクセル化後に判別可能か要スクショ検証
- **頭身厳密化が必要になった場合**: MakeHuman（CC0出力）で6.5頭身素体 → Blenderでデシメート＋フラットカラー化
- Humanoidボーンさえ整っていれば **素体は後から差し替え可能**。スパイクは無料素体で走り、
  絵作り確定後に差し替え判断
- 既製「野球選手モデル」は不採用（古い・品質バラバラ・色替え構造が不明でパラメトリック要件と不整合）

### 3.3 道具（自作）

- バット・グラブ・捕手ミット・帽子・ヘルメット・捕手防具は **自作プリミティブ**で足りる
  （ピクセル化後はシルエットのみ残る。バット＝先細り円柱、グラブ＝潰した楕円体レベルで本番品質）
- Blender Python（bpy）スクリプトで生成→FBX出力までClaude Codeに書かせる
- **アタッチ方式**: 手ボーンの子オブジェクトとして装着
  - `Animator.GetBoneTransform(HumanBodyBones.RightHand)` 等でTransform取得、位置・角度オフセットを一度調整
  - 全モーションに自動追従（スイング・バント・送球共通）
- 帽子/ヘルメットは別メッシュにして表示切替（打者=ヘルメット、守備=帽子）

### 3.4 左右ミラー

- 左打者・左投手は AnimatorState の **Mirror チェック**で生成（追加コストゼロ）
- **注意（CLAUDE.md反映事項）**: Mirror ON 時は道具のアタッチ先ボーンも左右反転が必要。
  「ミラー状態に応じてアタッチ先を切り替える」処理を1コンポーネントに集約すること

### 3.5 ユニフォーム色替え（設計書08の実装方式）

- 素体テクスチャを **16×16px程度のパレットテクスチャ**とし、UVを色ブロックに割り当てる
- マテリアルプロパティの差し替えのみで全チーム分を量産
- チームカラー定義 → マテリアル流し込みは、tokens.uss と同型のパターンとして部品辞書に登録

#### 3.5.1 色替え対象領域（6領域・確定）

| 領域 | プロパティ | 備考 |
|---|---|---|
| ユニフォーム（上） | 地色＋縦縞（有無/縞色） | 縞は3.5.3の手続き生成 |
| ズボン | 地色＋縦縞（有無/縞色） | 上下で別色可 |
| アンダーシャツ | 色 | 高校野球のチームアイデンティティの要 |
| 帽子（つば以外） | クラウン色 | |
| 帽子（つば） | つば色 | 2トーン帽子対応 |
| ソックス | 色 | |

#### 3.5.2 素体UV区分け要件（スパイク工程に追加）

- 色を分けたい単位でUV領域を最初から区切っておく必要がある:
  「アンダーシャツ＝前腕〜袖口」「ソックス＝すね〜足首」「つば＝帽子メッシュ内の別UV島」等
- 無料素体はこの区分けを持たないため、**bpyスクリプトで頂点グループ→UVブロック割り当て**を行う工程を
  スパイクに1手追加（道具生成スクリプト§3.3と同系統。Claude Codeの守備範囲）

#### 3.5.3 縦縞（模様）の実装方式

- 縞はテクスチャに焼かず、**シェーダプロパティで手続き的に生成**
  （縞ON/OFF・縞色・周波数の3プロパティ。マテリアル差し替え不要でパラメトリック制御）
- 縞の太さは **量子化後に縞1本≒1マクロピクセル、間隔2〜3マクロピクセル**を狙って周波数調整
  （細すぎる縞は量子化で消える／動くとモアレが出るため。元祖系ドット絵の縦縞と同じ太さ感）
- 3Dは体の曲面で縞が歪み、アニメで揺れるため、**最終判定はスクショ検証（§4.4）で行う**
- **フォールバック（縞がno-goの場合）**: 縞チームはユニフォーム地色を縞色寄りの中間色にして
  「縞っぽいトーン」で表現（縞の放棄ではなく段階的縮退）

## 4. ピクセル化シェーダ

### 4.1 二段構え（確定）

**ベースライン（自前・スパイクで実装）**

対決ビューはUI Toolkit埋め込みのため、どのみちRenderTexture経由で描画する（設計書06 §3.4系）。
これを利用する:

1. RTを低解像度（初期値 480×270）で生成
2. `FilterMode.Point` で拡大表示 → 解像度量子化が実質タダで手に入る
3. 色数制限: ポスタライズ or 3D LUT の軽量ポストパス1枚
4. 輪郭: 深度/法線エッジ検出1枚

全てURP Shader Graph＋Render Featureの教科書的構成。Claude Codeの守備範囲。

**品質オプション（go判定後に必要なら購入）: ProPixelizer（URP専用・数十ドル級）**

自前案との差分＝買う価値:

- オブジェクト単位のピクセル化（ピクセル化/非ピクセル化の混在可）
- オブジェクト単位のアウトライン（シルエット/法線ベース、色付け可）
- パレット・ディザパターンのエディタツール内蔵
- オルソ投影時のクリープレスなサブピクセル移動（低解像度RT方式の「動くとガタつく」弱点への対処済み）
- 画面自体はフル解像度維持のため、高解像度ポストとの併用可

導入時の注意: Graphics / Quality 両方のRenderPipelineAsset整合＋使用Rendererへの
Render Feature追加が品質段階ぶん必要（§2）。

### 4.2 絵作りパラメータ（初期値）

| 項目 | 初期値 | 根拠・備考 |
|---|---|---|
| RT解像度 | 480×270（FHDの1/4） | 比較枠: 320×180 / 640×360 |
| キャラの画面内サイズ | 高さ48〜64px目安 | 対決ビューでキャラが画面高5〜6割を占める構図前提 |
| 顔の実効ピクセル | 2〜4px | **顔なし方針の下限制約**。これを割る高解像度化は不可 |
| キャラの色数 | 8〜12色/キャラ | ユニフォーム2色＋肌＋道具。パレット制限はキャラのみに適用 |
| 背景の色 | 盤面「夏の昼間」トーンから直接サンプリングした固定パレット | 俯瞰⇄対決の往復で色を連続させる |
| 輪郭 | 1px相当・暗色（焦げ茶/ネイビー） | 純黒は夏昼の高明度背景に対し硬すぎる |
| **時間量子化（キャラのみ）** | ポーズ更新12fps相当（検証枠: 8 / 15fps） | 空間だけ量子化して時間を残すと「ドットのガワを被った3D」感が出る（§4.5-②）。**ボール・カメラはフルレート維持**（球筋の読みやすさ＝ゲーム性） |

### 4.3 UIテーマとの同居

- HUD・ストライクゾーン図・実況（スコアボード緑×アンバー）は **非ピクセル化のまま**
- 「世界＝ドット／計器＝クリーン」の対比は演出として正: スコアボードは"実物の計器"である
- 接続改善はフォントをドット系で揃える程度に留める

### 4.4 検証方法（既存体制への接続）

- バッチスクショ自己レビューに **RT解像度3段階 × 輪郭有無 × パレット有無 × 縦縞ON/OFF = 18枚マトリクス**を追加
  （縦縞ONは静止画に加え、スイング中のコマも撮ってモアレ/チラつきを確認する）
- 静止画に加え、**動画（投球1球ぶん）を時間量子化3段階（8/12/15fps）で書き出して比較**する
- 判定基準:
  1. 顔の不在が気にならないか
  2. 頭身がリアルに見えるか
  3. 盤面トーンと並べて違和感がないか
  4. 縦縞が量子化後も縞として読め、動いてもチラつかないか（§3.5.3の判定。no-goならフォールバック発動）
  5. 動きが「ぬるぬる」していないか＝コマ打ちの快感とモーキャップの自然さが両立しているか（§4.5-②）

### 4.5 モーション品質の懸念と対策（3件・方針確定）

3D＋モーキャップ購入方針に対して挙がった懸念と、その対処。いずれも方針変更を要しない。

#### ① 守備の捕球がバウンドの高低を無視して同じ芝居になる

- **現行スコープでは非発生**: 3Dキャラが映るのは対決ビューのみで、打球が出た瞬間に俯瞰（丸トークン）へ
  カットする（設計書06 §3.4）。ゴロを捌く野手の捕球芝居は現時点で画面に出ない
- 将来の3D俯瞰観戦（α後スパイク）で実弾化した場合の解法（二段構え）:
  1. **打球データ駆動のクリップ選択**: パックの捕球クリップは方向別5種（中/下/左/右/上）。
     エンジンのタイムラインが持つ打球座標から、捕球時点のボール高さでクリップを引く
  2. **グラブ手IK**: Unity Animation Riggingで、捕球直前の数フレームのみグラブをボール到達点に吸着。
     クリップ選択で大分類、IKで数十cmの端数を埋める

#### ② 「ぬるぬる動く3D」への拒否感（絵作りの核心）

- 原因: 空間（解像度・色数）だけ量子化して時間を量子化しないと、
  「ドット絵のガワを被った60fpsの3D」になり様式が崩れる
- 対策: **アニメーションの時間方向の量子化**——キャラのポーズ更新のみ12fps相当に間引く
  （2〜3コマ打ち。3Dでこれをやる先行例: スパイダーバース、Guilty Gear系）
- 実装: キャラごとにAnimatorのサンプリングを手動制御するコンポーネント1個（Claude Codeの守備範囲）
- **ボールとカメラはフルレートを維持**: 球筋の読みやすさはゲーム性なので量子化しない。
  「世界は滑らか・人間はコマ打ち」の使い分けが正
- パラメータは§4.2の表、検証は§4.4の動画比較で確定させる

#### ③ 捕手の捕球位置と投手の球筋が合わない

- ①の特殊ケースだが条件はより良い（エンジンは投球コース座標を保持、捕手捕球クリップは方向別5種）
- 解法（三段構え）:
  1. **コース→クリップのマッピング**: ストライクゾーン九分割相当から捕球クリップ5種へ対応付け
  2. **ミットIK吸着**: 捕球AnimationEventの直前0.1〜0.2秒のみIKウェイトを立ち上げ、
     ミット先端をボール到達座標に吸着
  3. **アンカー同期**: ボールはエンジンの球筋どおりに飛ばし、捕球時刻はタイムラインの
     アンカー逆算（§7-2の同期方式）で一致させる
- 大きく外れたコース（外角ボールゾーン等）で体ごと寄せる上半身Aimコンストレイントを足すかは
  検証スパイクで判断（§7-3のgo/no-go項目に含める）

     - 補足: §4.2のRT解像度3候補はFHD基準で整数倍が揃う組（320×180=×6 / 480×270=×4 / 640×360=×3）
     - B案（カメラviewport直描き＋透過パネル）は棄却: レイアウト手動同期・スクショ経路二系統化・z順制御難。
       ProPixelizer導入時もA案のまま「フル解像度RTに描いて埋め込む」に組み替え可能で将来を塞がない
   - **HUD同居: ゾーン重畳案**。ストライクゾーン図（配球・狙い球指示UI）を映像内の本塁上に半透明で直接重畳
     （TV中継のKゾーン風・非ピクセル化のチョーク白＝§4.3の「世界＝ドット/計器＝クリーン」と整合）。
     クリック座標⇄RT座標の変換が一手間必要（整数スケールスナップと同じ変換系に集約する）
2. **タイムライン同期**（論点4）
   - pitchセグメントの実時刻とモーション（足上げ→リリース）の同期
   - 同期方式の基本線: クリップにAnimationEvent（リリース/捕球/ミート）を注釈付けし、
     エンジン実時刻にEventが一致するよう**再生開始時刻を逆算するアンカー同期**。
     クリップ内部の緩急は改変しない（モーキャップの実演を壊さない）
   - 結果（空振り/ミート/見逃し）による打者モーション分岐
     → パックのMiss/Hitting別クリップ＋Batter_Strike（見逃し）にほぼ1:1対応する見込み
3. **検証スパイクの段取りとgo/no-go基準**（論点5）
   - スパイク構成案: 無料素体＋パック投球/スイング＋自前RT方式（出費はモーション代のみ）
   - go/no-go項目に追加: 時間量子化の快・不快（§4.5-②）、ミットIK吸着の成立（§4.5-③）
4. 横手投げ（サイドスロー）の扱い（オーバー・サブマリンは実弾あり。3種目をどうするか）
5. 素体の最終選定（Quaternius続投 or MakeHuman差し替え）— §4.4のスクショ判定後
6. 縦縞のgo/no-go（§3.5.3 / §4.4基準4）— no-goなら中間色フォールバックへ縮退
