# 設計書07: 実装ロードマップ（Claude Code開発）

設計フェーズの締め。原則: **エンジンが先、UIは後、3Dは最後**。
各フェーズに「完了条件（DoD）」を統計テストとして定義し、Claude Codeが自走で検証できる形にする。

---

## 1. リポジトリ構成

```
KokoSimNext/
├─ CLAUDE.md                  # Claude Code用ガイド（設計書索引＋不変条件＋規約）
├─ docs/design/               # 設計書01〜07（このシリーズをそのまま格納）
├─ engine/                    # 純C#ソリューション（Unity非依存 / .NET Standard 2.1）
│   ├─ KokoSim.Engine/        # 打席解決・育成・大会・イベント・監督メタ
│   ├─ KokoSim.Engine.Tests/  # 単体テスト＋統計回帰テスト
│   └─ KokoSim.Balance/       # ヘッドレス一括シミュレーションCLI
├─ data/                      # YAML（球種定義・イベント・校名語彙・球場・係数）
└─ unity/                     # Unityプロジェクト（UI Toolkit、将来3D観戦）
```

- エンジンはUnityへ**ローカルパッケージとして参照**（ソース共有）。`dotnet test` がUnityなしで回る
- バランス係数（設計書02の変換式など）はコードに埋めず `data/` のYAMLに置く

## 2. CLAUDE.md に書く不変条件

1. 二層構造: 打席解決は必ず物理層で計算する（表示値を直接確率に使わない）
2. 決定論: すべての乱数はシード付きRandomを注入。同シード同結果
3. エンジンにUnityEngine参照を持ち込まない
4. バランス変更は data/ の係数編集で行い、統計回帰テストを必ず通す
5. イベント・球種の追加はYAML編集のみで完結させる

## 3. フェーズ計画

| Phase | 内容 | 完了条件（DoD） |
|---|---|---|
| 0 | リポジトリ整備、CLAUDE.md、エンジン骨格、テスト基盤 | `dotnet test` がCIで回る |
| 1 | **打席解決**: 物理層変換・1球パイプライン・弾道積分（設計書01§2, 02§1-2） | 1万打席の打率/三振率/四球率/本塁打率が目標分布に収まる |
| 2 | **試合エンジン**: イニング進行、走塁・盗塁・バント・スクイズ、継投、投手スタミナ（02§1.1e） | 1万試合のスコア分布・試合時間が現実的 |
| 3 | **シーズンループ**: 週ターン、練習・育成・成長タイプ、合宿、イベント基盤（02§5, 03, 04§3-4） | CLIで10年自動プレイが無エラー完走、育成曲線が設計通り |
| 4 | **全国**: 学校生成、3層試合処理、大会構造、AI校運営（05） | 10年で勢力図が変動しつつ全国統計が安定 |
| 5 | **監督メタ**: 転任・信頼/名声・資金・スカウト/新入生（04§1-2） | 「修行編→フリー化」のキャリアが自動プレイで再現 |
| 6 | **Unity UI**: UI Toolkitへモックのテーマ移植。ホーム→選手→練習→試合(高速)→大会の順（06） | マウスで1シーズン遊べる |
| 7 | **詳細試合UI**: 投打対決ビュー→守備俯瞰の2ビュー構成（元祖踏襲）＋ポリッシュ | **プレイアブルα完成** |
| 8+ | エディタ、エクスポート対戦。3Dは**1週間スパイク**（カメラ＋ボール＋簡易球場のみ、人体なし）で継続判断 | — |

- Phase 1〜5 はUnityを一切開かない。**αの中身はすべてCLIで完成させる**
- シード固定リプレイはPhase 1から実装（バグ再現・将来のVTR機能の土台）

## 4. Claude Code運用プラクティス

- 各フェーズ開始時に該当設計書をコンテキストへ読み込ませてから着手する
- DoDの統計テストは「バランス回帰テスト」として常設し、係数をいじるたびに自動検証
- `KokoSim.Balance` CLIに `--report` を持たせ、打率分布などをMarkdownで出力 → 調整の議論に使う
- イベント量産・校名語彙・球場データは「YAMLを○件追加して」形式のタスクとして依頼
