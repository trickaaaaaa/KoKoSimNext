## 🔴 BREAKING — 選手モデル / 能力（設計書01・02）

1. **走塁系をスキルから連続パラメータへ移行**
   - 野手能力に `bunt`（バント）/ `steal`（盗塁）/ `baserunning`（走塁判断）の3つを**連続値パラメータ**として追加
   - 旧スキル「バント職人」「盗塁◎」等の走塁系スキルは**廃止**。該当スキル定義・参照を削除
   - 走塁/盗塁/バント/スクイズの解決式（設計書02 §4.1-4.4）をこの新パラメータ参照に変更
     - スプリント速度=走力、スタート遅延=走塁判断、盗塁の反応遅延=盗塁、バント成功率=バント主因

2. **「肩」を2パラメータに分割**
   - 旧: `arm`（送球速度＋精度）→ 新: `arm`（送球速度のみ）＋ `throwAccuracy`（送球精度）
   - 物理層変換に「送球精度→送球散布σ」を追加（設計書02 §1.2）。悪送球が中継・カバー連携を誘発

3. **運動能力ベースライン方式で選手生成を作り直す（設計書01 §1.1b）**
   - 生成を2段階に: ①身体能力ベースライン（地肩・バットスピード・脚力…専門非依存）を先に決める
     → ②専門能力を上乗せ。専門外能力も下限を持ち「できない」でなく「相対的に劣る」に収める
   - パワプロ型の極端特化（投手→野手能力壊滅／野手→投手不可）を禁止

4. **投打の左右を条件付き生成に（設計書01 §1.1c）**
   - 投げ手: 一般母集団 右88% / 左12%。投手層・強豪ほど左比率を上げる
   - 打ち手（投げ手からの条件付き）: 右投げ→右60%/左40%、左投げ→ほぼ左100%（左投げ右打ちは極稀）
   - スイッチは数%

5. **球速の生成分布を現実準拠の二層モデルに（設計書02 §1.1b、最速ベース）**
   - 一般母集団: 中央値120km/h台。甲子園級: エース140台がボリューム、二番手も140前後
   - 145+プロ注目 / 150+世代トップ / 155甲子園最速級 / 160+は「事件」（右裾を強く絞る）
   - ※旧実装が単純正規分布や高め平均なら要修正
   - **試合中の球速は毎球ばらつく（設計書02 §1.1d）**: 選手の球速値は最速（カタログ上限）。
     試合では最速を天井とした分布から1球ごとにサンプリング（常時は最速−3〜5km/h、力を込めた球で最速付近）。
     スタミナ疲労で終盤低下、調子・試合限りの好不調で上下、場面で最速。表示は最速・実況は実測球速

6. **球種ランクをストレートにも付与（設計書02 §2.1-2.2）**
   - ストレート含む全球種が G〜S ランク。内部2軸=球威＋キレ（表示はランク1本）
   - 球速(km/h)=生の速さ、ストレートランク=その球速の活かし度。「球速150だが棒球」等を表現

7. **野手の変化球レパートリー制限＋経歴由来の個人差（設計書01 §1.1b / 02 §2.2）**
   - 変化球はベースラインに含めない。野手はデフォルト変化球0〜1球種
   - ただし隠し属性「投手経歴」を確率的に付与。持つ野手は変化球保有数・質が上振れ
     （「中学までエース→高校で野手」「変化球多彩で投手もこなす野手」を稀少に存在させる）

8. **新入生生成を「総合力→凸凹配分」方式に（設計書04 §2.1）**
   - 名声が決めるのは**総合力（配る才能の総量）**であって各能力値ではない
   - 総ポイントを能力へ**ランダム重み付けで凸凹配分**（総合Cでも内訳B〜E混在）。配分の偏り自体もばらつかせる
   - 才能上限・伸びしろも分野別に凸凹配分。金太郎飴を禁止

8b. **身体系能力（走力・地肩・スタミナ）を名声と独立に生成（🔴 設計書01 §1.1b-2）**
   - 走力・地肩・スタミナは野球の上手さ（名声＝才能総量）と切り離してベースライン側で独立に振る
   - → 「弱小校の俊足」「弱小校の強肩」「弱小校の完投型」が自然に生まれる。機動力野球・盗塁の足場
   - ※旧実装が走力・肩を名声/総合力配分に含めているなら、身体系だけ別枠生成に要変更

8c. **地肩→球速/送球を二本立てに（🔴 設計書01 §1.1b-3 / 02 §1.2）**
   - 地肩（arm strength）を身体の土台に。野手の送球速度≒地肩、投手の球速=地肩＋投手センス補正
   - 「肩を鍛えれば球速が上がるだけ」を回避。地肩が同じでも投手練習の有無で球速が変わる（相関はあるが非連動）
   - ※旧実装が球速と肩を単一パラメータ/完全連動にしているなら分離が必要

9. **伸びしろ（分野別・成長効率倍率）を追加（設計書01 §1.1 / 02 §5.1）**
   - 投手/打撃/守備の分野別「伸びしろ」= 経験値効率倍率。現在値と独立
   - 経験値式に伸びしろ係数を乗算。天井は持たず、3年間の時間制約が実質到達点差を生む

10. **守備位置適性を可変に（設計書01 §1.1）**
    - 全9ポジション個別適性。練習・実戦で上昇。ポジション間の「近さ」で伸びの速さ・上限が変わる
      （内野内/外野内=易、内⇔外=難、捕手・投手=特殊で上限低め）

10b. **スタミナを等級管理→「目安投球数」の別管理に（🔴 設計書02 §1.1e）**
    - スタミナを他の投手能力ランクから切り離し、**目安投球数（例110球）で直接表示**＋内部は消耗カーブ
    - 他能力と独立に生成 → 「弱小校の完投型」「強豪の細いエース」が生まれる
    - 1球ごとに消耗し残量減で球威・制球低下（§1.1dの球速ばらつき・終盤低下を駆動）。球数制限・継投と連動
    - ※旧実装がスタミナをS〜Gランクで持っているなら数値（投球数）管理へ要変更

10c. **采配「飛ばしていけ/流していけ」ギア（🟢 設計書02 §1.1f / 09）**
    - 投手指示にギア設定を追加。飛ばす=スタミナ消費増の代わりに球速天井・球威↑（早くバテる）、
      流す=省エネで温存。場面指定可。エンジンはギア係数を球速サンプリングと消耗カーブに乗せるだけ

## 🔴 BREAKING / 🟢 NEW — 試合UIとエンジン出力契約（設計書06 §3.4 / 設計書01 §2・09）

**重要**: これは「見た目」でなく**エンジンが何を出力するかの契約**。既存の試合エンジンが結果だけ返す作りなら要修正。

32. **エンジンはタイムライン付きイベント列を出力する（🔴 出力形式の契約）**
    - 試合エンジンは各プレーを「誰が・どこから・どこへ・何時(t)に到達・何時に捕球/送球」という
      **時刻付きイベントのタイムライン**として出力する（座標＋時刻を持つ）
    - UIは物理計算を一切せず、時刻 t を進めて**再生するだけ**（判定・計算はエンジン側で完結済み）
    - この分離により、倍速・スロー・リプレイ・エラー・盗塁死・けん制・複数野手連携が
      すべて同一の仕組みで表現できる。後から3Dビューを乗せても既存作業が無駄にならない
    - ※既存実装がイベント列でなく最終結果のみ返すなら、タイムライン出力に拡張が必要
    - **暴投（ワイルドピッチ）・捕逸（パスボール）もタイムラインイベントとして扱う**: 専用ロジックでなく、
      投球コースの散らばり（コントロールσ）×捕手の捕球で自然発生。走者進塁の動きも俯瞰ビューで再生。
      暴投=投手責任（制球）、捕逸=捕手責任（捕球）の区別は投手σ×捕手捕球パラメータの合作で表現

33. **守備連携は「守備陣形ルール表」で自動生成（🟢）**
    - カバーリング・中継・ベースカバーを個別にハードコードせず、状況→各野手の担当動作を
      引く**ルール表**でタイムラインに動きを自動付与（9人全員が毎プレー役割を持つ）

34. **詳細モードは元祖踏襲の2ビュー構成（🟢 UI、該当フェーズで）**
    - 投打対決ビュー（1球指示UI: ストライクゾーン図で配球・狙い球＋作戦パネル、対決カットインの紙芝居）
    - 打球が出た瞬間に守備俯瞰ビューへカット（カット時刻=タイムラインの接触イベント）
    - 高速モード（スコアボード＋テキスト速報）も用意。画面中央は将来3Dに差し替え可能なコンテナに

35. **参照モック（視覚・データ構造の正）**: `mock-match-2d-view.html`（打球7プレー・9人守備）、
    `mock-steal-2d-view.html`（盗塁3パターン・9人守備・ベースカバー）。
    これらは「エンジンが出力する想定タイムラインをUIが再生する」構造の実証。UI実装時の参照仕様とする

## 確定した基本方針（再確認・全体を貫く原則）

- 「基本は実力どおり＋たまに番狂わせ」。運ゲー化しない範囲で揺らぎを入れる
- 「能力≠結果」を精神力・調子・慣れ・スキル・伸びしろの重なりで生む
- パラメータで出せるものはスキルにしない（二重計上禁止）
- 現実準拠を"風味の近似"でなく"実仕様準拠"で（球速分布・左右比率・47県形式）
- 顔グラなし（記号アイコン・背番号）。個性は数字・スキル・物語で立てる
