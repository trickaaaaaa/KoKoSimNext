## 0. 前提と方針

### 0.1 現行アーキテクチャの性質（重要）

試合エンジンは「野手一人ずつ追ってルール裁定する」型ではなく、
**打席結果を粗い列挙型 `PlateAppearanceResult`（三振/四球/単〜本/凡打/失策）へ落とし、
`BaserunningModel` が塁状態を合法遷移のみで進める**型。
（`engine/KokoSim.Engine/Match/AtBat/PlateAppearanceResult.cs` / `Match/Game/BaserunningModel.cs`）

帰結:
- **ルール違反（非合法な塁状態・アウト数不整合）は原理的に発生しない**。本書はその心配への対応ではない。
- 課題は逆方向 ——「本来あるプレーが確率バケットに吸収され個別表現されない / そもそも無い」という**網羅性**。

### 0.2 不変条件（本書の実装も従う）

- **#1 二層構造**: 追加プレーも表示能力値を直接確率に使わず、物理層／既存係数経由で解く。新しい確率は `data/coefficients.yaml`（`BaserunningCoefficients` 等）に集約する。
- **#2 決定論**: 追加イベントも `IRandomSource` を注入。乱数消費順を変えると既存帯がずれるため、**新イベントは既存の判定列の前後どちらに挿すかを明記**する。
- **#5 統計回帰**: 本書のプレーはすべて得点環境（出塁率・得点・失策・盗塁記録）を動かす。実装後は `data/balance-targets.yaml` の帯を再校正し、Heavy カテゴリの統計回帰を通す。

### 0.3 帯再校正の運用

1プレー入れるたびに帯を回すより、**P1 群をまとめて実装 → 一括で Heavy 回帰 → 帯を一度に再校正**が効率的（[[deferred-stats-work]] の方針）。
各プレーに「帯影響」欄を付す。影響ゼロ（既定オフで従来一致）を保てるものは、係数の既定値で無効化して段階導入する。

## 2. プレー別 実装仕様

各項: **現状 / あるべき / 実装方針（触る箇所）/ 係数（data）/ 乱数順 / 帯影響 / テスト**。

### P1-1. 野選（フィールダースチョイス）

- **現状**: `PlateAppearanceResult.InPlayOut` は打者を必ずアウト計上（`GameEngine.cs` の `batterOut`）。一塁走者ありのゴロは「併殺」か「打者アウト＋走者進塁」の二択で、**打者が生きて先行走者が刺される野選が無い**。
- **あるべき**: 一塁（or 封じられた塁）に走者がいるゴロで、一定確率「先行走者アウト＋打者は一塁生存（打数はアウト、安打ではない）」。
- **実装方針**:
  - `BaserunningModel.ApplyInPlayOut` に「打者生存フラグ」を返す経路を追加。現状の DP 判定（`DoublePlayProb`）の内側に、DP不成立時の分岐として `FieldersChoiceProb` を置く。
  - `GameEngine` 側で `batterOut` を戻り値で上書き（打者はアウトにせず一塁へ、先行走者を1アウト計上）。打者の打撃記録は「打数1・無安打」。
  - 打者走力が高いほど野選（DP崩れ）に寄りやすい: `DoublePlayProb` の走力補正を流用。
- **係数**: `BaserunningCoefficients.FieldersChoiceProb`（DP不成立かつ一塁走者ありの残余から按分）。
- **乱数順**: DP判定の直後に1回追加（DP成立時は消費しない＝既存の DP 経路の乱数は不変）。
- **帯影響**: 打者の出塁がわずかに増える一方アウトは変わらない。BABIP・出塁率の帯を微調整。
- **テスト**: 一塁走者あり無死で多数試行し、DP率＋野選率＋通常凡打率＝1 に収束。打者の安打計上がゼロ。

### P1-2. 振り逃げ（第3ストライク不捕球）

- **現状**: `PlateAppearanceResult.Strikeout` は無条件アウト、走者進塁なし。
- **あるべき**: 一塁が空 or 2アウトの三振時、低確率で捕手が捕り損ね、打者走者が一塁を狙う。成功＝出塁（投手には三振を計上、打者は打数1・無安打・アウトにしない）。捕手が拾って一塁送球アウトも有り。
- **実装方針**:
  - `AtBatResult` に「不捕球フラグ」を持たせるか、`GameEngine` の三振処理直後に判定を挿す。適用条件は現実ルール準拠（`bases.First == null || outs == 2`）。
  - 成否は捕手 `Fielding/Catching` と投球の質（落ちる球ほど逸らしやすい）で決まる。物理層を持たないので当面は捕手能力ベースの低確率テーブル。
  - 出塁時は塁状態を「打者一塁到達＋（他走者は原則進まず、暴投扱いなら P2-8 と連動して進む）」。
- **係数**: `BaserunningCoefficients.DropThirdStrikeReachProb`（基準）＋捕手補正係数。
- **乱数順**: 三振確定後・イニング後処理前に挿入（三振以外の打席結果では消費しない）。
- **帯影響**: 三振の一部が出塁へ。K/9 は不変、出塁率が微増。
- **テスト**: 一塁ふさがり0/1アウトでは発生しないこと。空き塁 or 2アウトでのみ低頻度で出塁が発生。

### P1-3. 故意四球・敬遠（申告敬遠含む）

- **現状**: 守備側采配（`ITacticsBrain.CallDefense`）に敬遠の選択肢が無い。
- **あるべき**: 一塁が空き・強打者・得点圏など条件下で守備側が敬遠を選択→打者は無条件四球（`Walk`、押し出し進塁は既存 `ApplyWalk`）。次打者勝負の采配。申告敬遠は投球数0でトグル、通常敬遠は4球（`TotalPitches` に加算）。
- **実装方針**:
  - `DefensiveTactics` に `IntentionalWalk` フラグ（または `ITacticsBrain.CallIntentionalWalk`）を追加。`GameEngine` の打席解決前に判定し、真なら `AtBatResolver` を通さず `PlateAppearanceResult.Walk` を直接確定。
  - 敵AI（`AiTacticsBrain`）: 校風「守り勝つ」・強打者（Power/Contact 上位）・一塁空き・僅差終盤で確率上昇。プレイヤーは手動指示。
  - 統計上は四球に含める（記録を分けたい場合のみ `IntentionalWalk` を別列挙）。`ModernRules` に申告敬遠トグル（年代連動、設計書05 §1.3）。
- **係数**: `TacticsCoefficients.IntentionalWalk*`（敵AIの発動閾値）。`ctx.ModernRules.AutoIntentionalWalk`（申告敬遠の可否）。
- **乱数順**: 打席解決の前段。発動時は `AtBatResolver` の乱数を消費しない（消費順が変わるので、既定オフ＝敵AI未設定時は分岐に入らないこと）。
- **帯影響**: 強打者の四球増・後続打者の対戦増。押し出し・満塁策の得点分布。
- **テスト**: 一塁走者ありでは敬遠しない（押し出しリスク）。強打者×一塁空きで発動率が上がる。

### P1-4. ダブルスチール／二盗・三盗・本盗

- **現状**: `StealResolver` と `GameEngine` の盗塁分岐は**一塁走者・二塁空の単独二盗のみ**（`bases.First is not null && bases.Second is null`）。
- **あるべき**: 走者一・二塁 / 一・三塁 / 二・三塁の重盗、単独三盗、（終盤の）本盗。各走者の走力・走塁判断と捕手肩・送球先選択で解決。
- **実装方針**:
  - `StealResolver` を「走者集合＋捕手＋送球先選択」を取る多重盗対応に一般化。捕手はどの塁へ投げるか（先行 or 後続）を選び、投げた塁だけがアウト対象。
  - `GameEngine` の盗塁分岐を単独二盗前提から、`OffensiveSign.Steal` の対象走者を状況で決める形へ拡張。本盗は三塁走者＋スクイズ/重盗の文脈で低頻度。
  - 采配 `ITacticsBrain.CallOffense` が重盗を選べるよう `TacticsSituation` は既存で十分（走者3塁分を保持済み）。
- **係数**: 既存 `BaserunningCoefficients` の盗塁成功式を流用。三盗は成功率下方、本盗はさらに下方の基準。
- **乱数順**: 打席前の盗塁ブロックを拡張（既存の単独二盗と同じ位置）。**単独二盗の乱数消費は現状と一致させる**（重盗分岐は走者構成が異なる時のみ入る）。
- **帯影響**: 盗塁企図・成功/刺殺の記録。機動力校の得点。
- **テスト**: 一・二塁で重盗の成功時に両走者が1つずつ進む。捕手が二塁送球を選ぶと一塁走者だけ危険、等の分岐が決定論で再現。

### P1-5. 牽制アウト／離塁刺殺

- **現状**: 走者が刺されるのは盗塁死のみ。牽制・離塁の概念が無い。
- **あるべき**: 盗塁企図の裏で、投手のクイック・牽制と走者のリード（走力・走塁判断）で低確率の牽制アウト。飛び出しの大きい走者ほど危険。
- **実装方針**:
  - 打席前の盗塁ブロック手前に「牽制チェック」を低確率で挿入（`StealResolver` の副次、または `PickoffResolver` 新設）。
  - 発動は主に盗塁を狙う走者・リードの大きい俊足走者に限定し、通常局面では確率ほぼ0（既定オフに近い係数）。
- **係数**: `BaserunningCoefficients.PickoffBaseProb`（極小）＋走者/投手補正。
- **乱数順**: 盗塁判定の直前に1回。既定係数0なら消費しても結果不変になるよう設計（あるいは企図フラグがある時のみ判定）。
- **帯影響**: 盗塁企図のリスク側。盗塁成功率をやや押し下げる方向で帯調整。
- **テスト**: リード最小の走者では発生率ほぼ0。盗塁多用チームで牽制死が現れる。

### P1-6. エラーの連鎖（失策での追加進塁・生還）

- **現状**: `PlateAppearanceResult.ReachedOnError` は `ApplyRunnerAdvance(isDouble:false)` で**単打相当・打者一塁**。失策で走者が余分に進む/生還する、失策で二塁到達、といった連鎖が無い。
- **あるべき**: 失策時に一定確率で「走者が通常より1つ多く進む」「打者走者が二塁まで（送球エラー）」。悪送球の連鎖。
- **実装方針**:
  - `ReachedOnError` 専用の進塁処理を分岐（現状は Single と共通）。基準進塁に加え `ErrorExtraAdvanceProb` で全走者＋打者に追加1個の判定。
  - 失策の記録（`defense.AddError`）は既存。追加送球エラーは失策数を追加計上するか記録簡略化かを決める（当面は追加失点のみ、失策数は1のまま簡略で可）。
- **係数**: `BaserunningCoefficients.ErrorExtraAdvanceProb`。
- **乱数順**: `ReachedOnError` の進塁内で追加消費（Single と経路を分けるため、Single 側の乱数順は不変）。
- **帯影響**: 失策絡みの失点が増える。失点分布・被失策時の期待得点。
- **テスト**: 失策多発設定で「失策1つあたりの平均進塁」が基準を上回ることを確認。

#### P1-6b. 送球精度（ThrowAccuracy）の接続（Issue #37, 2026-07-23）

送球精度 `ThrowAccuracy` は変換式・生成・成長まで実装済みだが守備解決から未参照だった。悪送球の発生を
送球者の能力で動かす。写像方式は**簡易確率式（能力値直接・σ非経由）**を採用（決定点A=(a)）。

- **失策連鎖のσ→能力駆動化**: 連鎖確率 = `ErrorExtraAdvanceProb − (Ac−50)×ErrorExtraAdvanceAccuracySlope`
  （送球者＝失策を犯した野手の `ThrowAccuracy`。clamp は既存 min/max を流用）。**Ac=50 で現行の一律確率と恒等**
  （傾き項=0）＝リーグ平均で校正値・乱数消費順とも不変（帯不変, 不変条件#5）。傾き既定は 0.0（＝現行と完全一致）、
  YAML で正値を与えると精度連動が有効化（`ErrorExtraAdvanceProb` の sim/Unity 分離と同じ運用）。
- **内野ゴロ→一塁送球の按分（2段階, 決定点B）**: 捕球ロール（`Catching`, 現行 `MaybeError`）→ 送球ロール
  （`ThrowAccuracy`）の順で判定する。送球ロールの失策確率 = `ThrowErrorBaseProb − (Ac−50)×ThrowErrorAccuracySlope`。
  `ThrowErrorBaseProb` 既定 0.0（＝送球ロールを一切引かない＝現行の捕球のみ判定と乱数消費順・結果とも完全一致）。
  正値を与えると送球エラーが加算され、`ThrowAccuracy` が低いほど内野ゴロが安打/失策になりやすくなる。捕球エラー
  （`ErrorBaseProb`, Catching）と送球エラー（`ThrowErrorBaseProb`, ThrowAccuracy）は独立ロールで、
  低い方が悪いと悪送球になる二段関門。空中捕球（フライ）は送球を伴わないため従来どおり捕球ロールのみ。
- **係数（`data/coefficients.yaml`）**: `fielding.throw_error_base_prob` / `throw_error_accuracy_slope`、
  `baserunning.error_extra_advance_accuracy_slope`。傾き・下限は #4 に従い YAML 駆動。

> **実ゲーム発火（Issue #169, 2026-07-23確定）**: #37 の配線を実ゲームで有効化。送球ロールと失策連鎖を
> **C#既定＋YAMLの両方**で正値化（Unity実プレーは `new GameContext()` 既定で動くため、sim/Unity係数分離に従い両方必須）。
> - `throw_error_base_prob`: 0.0 → **0.0266**（Ac50 で内野ゴロ失策の**約40%が送球由来**・残り約60%が捕球由来）
> - `error_base_prob`: 0.064 → **0.0384**（送球ぶんを捕球側から再配分。総失策の母平均は≈3.4→≈2.9で甲子園実測寄りに改善）
> - `error_extra_advance_prob`: C#既定 0.0 → **0.20**（YAML既存値に一致）＝失策連鎖を実ゲームでON
> - `error_extra_advance_accuracy_slope`: C#既定 0.0 → **0.003**（YAML既存値に一致）＝連鎖の送球精度連動をON
>
> 2段階化で rng 消費順が変わるため**決定論baseline（determinism-baseline.txt）を再生成**。Heavy統計回帰で
> 失策数/試合・失策連鎖・BABIP・内野安打の全帯内を確認。ThrowAccuracy の高低で失策/悪送球が実ゲームでも単調に変わる。

### P2-7. 死球（HBP）

- **現状**: 列挙型に無し。四球のみ。
- **あるべき**: `PlateAppearanceResult.HitByPitch`。打者一塁へ（`ApplyWalk` と同じ押し出し進塁）、打数に数えない、四球とは別記録。荒れ球投手（`Skill.EffectivelyWild`）・インハイ攻めで増。
- **実装方針**:
  - 列挙型追加。`AtBatResolver` の四球判定の近傍で、コントロール散布（`ControlScatter`）が大きく外れinした場合に低確率で HBP。二層に沿い、制球のσから物理的に導出できると理想（当面は制球ベースの低確率）。
  - `IsAtBat` は false（四球と同様）、`IsHit` false。成績表示は四球と分離。
- **係数**: `BattingCoefficients.HitByPitchProb`（制球依存）。`Skill.EffectivelyWild` で増。
- **乱数順**: 四球確定の近傍。既定係数0で従来一致を保てる形に。
- **帯影響**: 出塁率の四死球内訳。全体出塁はわずか増。
- **テスト**: 制球極悪投手で HBP 率が上がる。打数に加算されない。

### P2-8. 暴投・パスボール

- **現状**: 走者ありの投球で暴投/捕逸による進塁が無い。
- **あるべき**: 走者ありの各投球（特に落ちる球・2ストライク）で低確率の暴投/パスボール→全走者1つ進塁、三塁走者は生還。振り逃げ（P1-2）と機構を共有。
- **実装方針**:
  - 打席解決の投球ループ内 or 打席前に「バッテリーミス」低確率イベント。捕手 `Catching` と投球の質で確率。発生時は `BaserunningModel` に軽量な「全走者1つ進む」処理を追加。
  - 暴投=投手責、パスボール=捕手責で記録を分けられると良いが、当面は進塁のみで記録簡略可。
- **係数**: `BaserunningCoefficients.WildPitchProb` / `PassedBallProb`（走者ありのみ）。
- **乱数順**: 走者ありの打席のみで判定を追加（走者無しでは消費しない＝走者無しの帯は不変）。
- **帯影響**: 走者進塁・小技得点が増。得点の底上げ。
- **テスト**: 走者三塁で暴投による生還が低頻度で出る。走者無しでは一切発生しない。

### P2-9. ライナー併殺（飛び出し憤死）

- **現状**: 併殺は「ゴロ＋一塁走者」型のみ（`DoublePlayProb`）。ライナー直撃で走者が戻れず刺される併殺が無い。
- **あるべき**: 打球種別がライナー（`InPlayOut`）で走者ありのとき、低確率で「捕球＋走者飛び出しアウト」の併殺。
- **実装方針**:
  - `AtBatResult.Play`／`BattedBall` の打球種別（ゴロ/ライナー/フライ）を `BaserunningModel` へ渡す。ライナー凡打かつ走者ありで `LineDriveDoublePlayProb` を判定。
  - 打球種別はすでに物理層（打角・初速）から出るので、二層に沿って分類できる。
- **係数**: `BaserunningCoefficients.LineDriveDoublePlayProb`。
- **乱数順**: `ApplyInPlayOut` 内、ゴロ併殺判定とは打球種別で排他（ライナー時のみ消費）。
- **帯影響**: 併殺の内訳。全体アウト分布は微変。
- **テスト**: 打角の高いライナー凡打・走者ありでのみ発生。ゴロでは従来のゴロ併殺のまま。

### P2-10. 満塁ゴロの本塁封殺

- **現状**: 併殺は二塁封殺型のみ。満塁ゴロで本塁フォースアウト（本封→一封の併殺、本封単独）が無い。
- **あるべき**: 満塁（or 三塁が封じられる状況）のゴロで、内野が本塁封殺を選ぶ分岐。内野の深さ（`DefenseDepth`、G1 と連動）で「本塁封殺 vs 一塁アウトで1点献上」を選択。
- **実装方針**:
  - `ApplyInPlayOut` の三塁走者処理に、満塁＋内野前進時の「本塁フォースアウト」経路を追加（`extraOuts++`、三塁走者アウト、得点0）。深い守備なら従来どおり献上して一塁アウト。
- **係数**: 既存の `HomeGrounderStartDelaySeconds` / 深さ判定（G1）を流用。新規最小。
- **乱数順**: 三塁走者判定内。満塁かつ前進守備時のみ分岐（他は不変）。
- **帯影響**: 満塁時の失点分布（0点封殺 vs 1点）。
- **テスト**: 満塁・内野前進・無死/一死で本塁封殺が出る。二死では打者アウトでチェンジ。

### P3-11. 犠飛以外のタッグアップ（二塁→三塁）

- **現状**: 深いフライ後の二塁走者三進は `ProductiveOutAdvanceProb` に吸収（タッグアップ個別判定でない）。
- **あるべき**: フライアウト時、二塁走者が飛球の深さ・走力でタッグアップ三進。犠飛（三塁走者）と同じ機構の拡張。
- **実装方針**: `ApplyInPlayOut` のフライ分岐で、打球の飛距離/滞空（`FieldingPlay`）を使い二塁走者のタッグアップを個別判定。ゴロでは従来の進塁打のまま。
- **係数**: `BaserunningCoefficients.TagUpSecondToThirdProb`（滞空・走力依存）。
- **帯影響**: 走者進塁の精度向上。小。
- **テスト**: 深い外野フライで二塁走者の三進率が上がる。浅いフライでは進まない。

### P3-12. 併殺の型の多様化（5-5-3 / 6-6-3 等）

- **現状**: 判定は正しいが、タイムライン演出（設計書12 F1）が **6-4-3 型固定**（`TimelineBuilder.AppendDoublePlayThrows`）。
- **あるべき**: 打球方向（三塁側ゴロなら 5-4-3 / 5-5-3、遊撃 6-4-3 / 6-6-3、一二塁間 3-6-3 / 4-6-3 等）で送球連鎖の演出を変える。
- **実装方針**: **判定・帯には一切干渉しない純演出**。`FieldingPlay` の処理野手・打球方向から送球連鎖のノード列を選ぶ。`TimelineBuilder` にパターンを追加。
- **帯影響**: **なし**（表示専用）。統計回帰不要。
- **テスト**: タイムラインの送球ノード列が打球方向に対応（スナップショット的検証）。

### P3-13. ランニングHR／場内本塁打

- **現状**: 本塁打は柵越え相当。場内本塁打は三塁打＋走者生還に吸収。
- **あるべき**: 広い球場（設計書13）で外野を抜ける打球が三塁打止まりでなく生還しきる稀ケース。頻度極小。
- **実装方針**: `ApplyRunnerAdvance` の三塁打分岐で、打球の飛距離・球場寸法・打者走力から低確率で打者本塁生還（=本塁打記録 or 場内本塁打）。**優先度最低・当面据え置き可**。
- **帯影響**: ほぼ無し。
- **テスト**: 広い球場・俊足でのみ極稀に発生。

## 2.5 現代ルール（ModernRules）拡張

プレー（局面で起きる動作）とは別軸の、**現代ルール**の層。

> **前提: プレイは2026年スタート固定。** よって2026年時点で施行済みの現代ルールは**すべて常時ON（ベースライン）**でよい。
> `ModernRules`（`engine/KokoSim.Engine/Nation/ModernRules.cs`）の年代分岐（`*IntroYear`）は `year=2026` で全ルールが自動ONになるため、
> **過去年代・「元祖の時代」への切替は基本ゲームでは実装不要**（`*ForcedOff` トグルは将来のフレーバー用に残置するだけ）。

現状の `ModernRules` は **DH / タイブレーク / 球数制限** の3つを持ち、`year=2026` で3つとも自動ON＝**追加実装は不要**（確認のみ）。
未実装の以下を「2026年ベースラインで常時ON」として足す。

|---|---|---|---|
| DH / タイブレーク / 球数制限 | ✅実装済（2026で自動ON） | — | 確認のみ |
| **申告敬遠（自動敬遠）** | ❌ | 小（敬遠の投球数0化） | 采配（P1-3 と接続・常時ON） |

| クーリングタイム / 継続試合 | ❌ | 小（疲労回復・運用） | 運用（当面スキップ可） |

### 2.5-a. 申告敬遠（自動敬遠）

- **内容**: 2026年ベースラインでは申告制＝投球せず宣言のみで四球成立（`TotalPitches` に加算しない）。P1-3 の敬遠処理と一体で、常に申告制として扱う。
- **実装**: P1-3 の敬遠が発動したら投球数0で `Walk` 確定。年代分岐は不要（2026固定なので常に申告制）。
- **帯影響**: 投球数・試合時間がわずかに減る程度。得点への直接影響は小。

### 2.5-b. 低反発金属バット → 不採用（普通のバット基準）

- **判断（2026-07-17）**: 低反発バットは本塁打・長打が減り**ゲームとして面白みが下がる**ため、**採用しない**。打球物理のベースラインは従来どおり「普通の（よく飛ぶ）金属バット」とする。
- **帰結**:
  - 打球物理・`balance-targets.yaml` の本塁打率/長打率の帯は**現行のまま**。低反発向けの再校正・専用帯・`BatExitVeloFactor` トグルは**いずれも不要**。
  - 現実の高校野球ルールとは意図的に乖離させる（ゲーム性優先。「元祖に忠実＋現代化」のうち、この点は面白さを取る）。
- **将来余地**: どうしても後で「低反発モード」を足したくなった時だけ、係数 `BatExitVeloFactor`（<1.0）をオプションで導入する。標準はあくまで普通のバット。

### 2.5-c. コリジョンルール（本塁での接触プレー制限）

- **内容**: 危険な本塁突入・ブロックの制限。2026年ベースラインで常時ON。本塁クロスプレー（設計書12 F2 / `HomePlayResolver`）の**タッチ成立/送球判定**を走者側やや有利に補正（捕手のブロック不可）。
- **実装方針**: `HomePlayResolver` の成功確率に常時の小補正を織り込む（年代分岐なし）。校正後の帯に含めて回帰。
- **帯影響**: バックホーム憤死の頻度が微減。統計参考値（`HomePlayOuts`）に反映。校正時に一緒に吸収。
- **テスト**: 本塁生還率が補正後の帯に収まる。

### 2.5-d. クーリングタイム / 継続試合（当面スキップ）

- 猛暑対策の休憩・日跨ぎ継続などの運用ルール。試合結果への影響は小さく、**優先度最低**。（選手疲労は廃止済のため疲労管理とは絡めない）

## 3. 実装順序と帯再校正計画

段階的に、各段の末尾で Heavy 統計回帰＋帯再校正を1回。

1. **第1段（P1コア・出塁と采配）**: 野選 / 振り逃げ / 敬遠 / 重盗・三盗 / 牽制 / 失策連鎖。
   → 得点環境が最も動く。実装後に `balance-targets.yaml` を一括再校正。
2. **第2段（P2・小技と細部）**: 死球 / 暴投・パスボール / ライナー併殺 / 満塁本封。
   → 走者進塁・併殺内訳の微調整。再校正2回目。
3. **第3段（P3・精度と演出）**: タッグアップ二→三 / 併殺演出多様化 / ランニングHR。
   → 演出中心（12は帯影響なし）。帯影響は小。

**現代ルール（§2.5）— 2026年固定なので全て常時ON。年代分岐は不要**:
- DH / タイブレーク / 球数制限は実装済（2026で自動ON）＝**確認のみ**。
- **申告敬遠**は第1段の敬遠（P1-3）と同時。常に申告制。
- **低反発バットは不採用**（§2.5-b）。普通のバット基準のまま＝打球物理・帯は現行維持、追加作業なし。
- **コリジョンルール**は第2段（本塁プレー系）と併せ、常時ONの小補正として校正に含める。

各プレーは**係数の既定値で無効化した状態から入れ、既定オフでは従来帯と完全一致**を最初のテストで担保する（設計書09/10 と同じ「無指示なら従来一致」方式）。その後に係数を有効化し帯を合わせる。
